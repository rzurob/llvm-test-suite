Overview

%Note{A brief description of the feature, its application(s) and limitation(s).

Feature syntax may also be mentioned here. Since it is in the I0 document, it is optional.}%

  The current system of interoperability between C and Fortran, defined in F2008 standard, does not provide means for interoperability to a C pointer to void. 
 
  In TS 29113 Further Interoperability of Fortran with C, the interoperability facilities have been extended to allow C procedures with void * parameters to interoperate with Fortran procedures through the newly introduced assumed type objects. The support for this feature is guarded by a langlvl option.

Testing will be done to ensure the proper use of the newly introduced assumed type objects. Assumed type object can only appear as dummy argument of either a BIND(C) procedure or a Fortran procedure. An assumed-type object is unlimited polymorphic and there is no mechanism to determine the actual type of an assumed-type entity.

This feature will not cover assumed type entities that are assumed rank and assumed shape.

Functional and diagnostic test cases will be created for this feature covering the following items:

Diagnostic Testing:


Positive diagnostic: make sure assumed type objects are accepted where the TS allows it: 

  • Assumed type dummy argument accepted for procedures with and without BIND(C) without -qlanglvl and with -qlanglvl set to extended or ts
  • Assumed type dummy argument accepted for procedures with and without BIND(C) with -qlanglvl set to other than extended or ts ==> an (L) level message emitted
  • Ensure that we give the right information for the assumed type object in the listing file with -qattr
  • Assumed type object can only be a dummy argument
  • Assumed type object as an actual argument that is argument associated with an assumed type dummy argument
  • Assumed type object as the first argument of IS_CONTIGUOUS, LBOUND, UBOUND, PRESENT, SHAPE, RANK, SIZE and C_LOC 

Negative diagnostic: diagnose and emit error message when assumed type appears where it is disallowed by the TS
  • Assumed type objects cannot appear in a print statement 
  • Assumed type objects cannot appear in an assignment statement
  • Assumed type objects cannot appear as an argument of an intrinsic procedure other than the ones listed above (Addition, Multiplication, MAX, MIN, ...)
  • Assumed type objects  cannot be declared as a local or module variable with or without the bind(c) or save attributes
  • Assumed type object as an actual argument that is argument associated with a dummy argument that is not assumed type
  • * and TYPE(*) cannot appear as the type-spec in an ALLOCATE statement.
  • TYPE(*) can't appear as the type-spec in an array constructor
  • Assumed type object cannot appear as a selector of a SELECT TYPE statement 
  • TYPE(*) cannot appear as the type guard statement in a SELECT TYPE construct
  • Assumed type object cannot appear as the selector in an ASSOCIATE statement 
  • Test that:  An assumed-type entity shall be a dummy variable that does not have the
    • ALLOCATABLE
    • CODIMENSION
    • INTENT(OUT)
    • POINTER
    • VALUE
    • explicit-shape array
  • Assumed type dummy argument cannot be argument associated with an actual argument that is of derived type if it
    • has type parameters
    • a type-bound procedure
    • or a final subroutine

Functional Testing :

1). The test cases will cover the following cases:

  • Calling from C a BIND(C) procedure defined in Fortran
  • Calling from Fortran a BIND(C) procedure defined in C
  • Calling from Fortran a BIND(C) procedure defined in Fortran
  • Calling from Fortran a non-BIND(C) procedure defined in Fortran

We will use nested calls and/or multiple arguments to verify that the type and type parameters of the dummy argument are properly obtained from the effective argument. We will explicitly provide additional information to the procedure in order to determine the type and the size of the object. The called function can modify the passed object through void * pointer to ensure the address of the correct object is passed.

2). Test that the assumed type dummy arguments can be one of the following:
  • Scalar assumed type entity
  • Assumed type entity that is an assumed size array (1 or multiple dimensions)
  • Assumed type entity with OPTIONAL attribute
  • Assumed type entity with CONTIGUOUS attribute
  • Assumed type entity with VOLATILE attribute
  • Assumed type entity with TARGET attribute

3). Test that the actual argument associated with the assumed type dummy arguments can be one of the following:

  • Of interoperable intrinsic type (see table below)
  • Of derived types
  • polymorphic object with a dynamic type that is interoperable with C
  • unlimited polymorphic object with a dynamic type that is interoperable with C
