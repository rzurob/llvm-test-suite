Feature Overview

We introduce assumed-type objects in this feature.

An entity declared with TYPE (*) is an assumed-type entity. It has no declared type and its dynamic type and type parameters are assumed from its effective argument.
An assumed-type object is unlimited polymorphic.

Key Requirements

None.

Key Design Issues

1. Should we diagnose TYPE(*) arguments for procedures that don't have BIND(C) ?

Key Design Decisions

1. TYPE(*) without BIND(C) is allowed. An assumed-type entity represents an address.

Testing Strategy

Development will provide unit tests while the FVT testing will focus on more in depth testing involving several features.

-- Unit Testing:

- Functional:
1. Test assumed-type object which is associated with an effective argument of intrinsic type.
2. Test assumed-type object which is associated with an effective argument dummy argument of derived type.
3. Test assumed-type object which is associated with an effective argument dummy argument of polymorphic object.
4. Test argument presence of an assumed-type dummy that has the OPTIONAL attribute.
5. Test miscellaneous items.

- Diagnostic:
1. Test miscellaneous diagnostic situations described in "Constraints" section in detail below.

-- FVT testing will consist of both disgnostic and functional tests:

Diagnostic:

the focus will be on the diagnostic cases that are not present in the unit tests. We need to make sure that the compiler diagnoses and properly flags any situation where the use of assumed type is not allowed by the specification.

1. Assumed Type object + Explicit shape array
2. Assumed Type object + Assumed size array
3. Assumed-type object as selector of either SELECT TYPE or ASSOCIATE statement.
4. TYPE(*) as the type guard statement is a SELECT TYPE construct

 



- Functional:
1. Test assumed-type object associated with an effective argument of intrinsic type.
2. Test assumed-type object associated with an effective argument of derived type.
3. Test assumed-type object associated with an effective argument of polymorphic object.
4. test optional assumed type dummy argument
5. with and without BIND(C)

- Diagnostic:


Functional:
1. calling from C, where the procedure is defined in Fortran (with bind(c) )
2. calling from Fortran, where the procedure is defined in Fortran (no bind(c) )
3. calling from Fortran, where the procedure is defined in C (with bind(c) )

- The feature won't have any appreciable impact on performance, then high optimization option testing is unnecessary.
User Documentation

Overview

We introduce assumed-type objects(TYPE(*)) in this feature.

User Benefits

The feature facilitates interoperability with formal parameters of type "void *" in C language. 

Public Interface

 
declaration-type-spec is intrinsic-type-spec
                         or TYPE ( intrinsic-type-spec )
                         or TYPE ( derived-type-spec )
                         or CLASS ( derived-type-spec )
                         or CLASS ( * )
                         or TYPE ( * )

Constraints

1. An assumed-type object that is not assumed-shape and not assumed-rank is intended to be passed as the C address of the object.

This means that there would be insufficient information passed for an assumed-type explicit-shape array that is an actual argument corresponding to an assumed-shape dummy argument. Therefore TYPE(*) explicit-shape is not permitted.

2. An assumed-type entity shall be a dummy variable that does not have the ALLOCATABLE,
CODIMENSION, INTENT(OUT), POINTER, or VALUE attribute and is not an explicit-shape array.

3. An assumed-type variable name shall not appear in a designator or expression except as an actual argument corresponding to a dummy argument that is assumed-type, or as the first argument to any of the intrinsic and intrinsic module functions IS CONTIGUOUS, LBOUND, PRESENT, RANK, SHAPE, SIZE, UBOUND, and C_LOC.

 

4. An assumed-type actual argument that corresponds to an assumed-rank dummy argument shall be
assumed-shape or assumed-rank.

5. An assumed-type dummy argument shall not correspond to an actual argument that is of a derived type that has type
parameters, type-bound procedures, or final subroutines.

Rules

An assumed-type object is unlimited polymorphic. And there provides no mechanism for a Fortran procedure to
determine the actual type of an assumed-type argument.

Messages

Samples

 

$ cat example1f.f
use, intrinsic :: iso_c_binding
implicit none
integer(c_short)      :: i_short
integer(c_int)        :: i_int
integer(c_long)       :: i_long
integer(c_long_long)  :: i_long_long

interface
   subroutine c_func(a, flag) BIND(c)
      use, intrinsic :: iso_c_binding
      implicit none
      type(*) :: a
      integer(c_int), value :: flag
   end subroutine c_func
end interface

i_short = 2
i_int = 4
i_long = 4
i_long_long = 8

call c_func(i_short, 1)
call c_func(i_int, 2)
call c_func(i_long, 3)
call c_func(i_long_long, 4)

end


$ cat example1c.c
#include <stdio.h>

void c_func(void* a, int flag)
{
   if(flag == 1)
      printf("Type is c_short %hd\n",*(short*)a);
   else if(flag == 2)
      printf("Type is c_int %d\n",*(int*)a);
   else if(flag == 3)
      printf("Type is c_long %ld\n",*(long*)a);
   else if(flag == 4)
      printf("Type is c_long_long %lld\n",*(long long *)a);

  return;
}
