Overview

Description

Technical Specification 29113 allows assumed-length character type variable to be interoperable with C in the Fortran language. An assumed-length character variable is a dummy argument whose length is assumed from its actual argument at run-time. Such an argument is intended to facilitate interoperability with C functions. The corresponding C formal parameter is a CFI_cdesc_t rather than character type.


Related Features

C Interop: Assumed Rank

Feature Details

The assumed-length dummy argument has the form of "character(*) :: arg" or "character(*) :: arg(array_spec)". It means the character length is assumed from the actual argument.

The feature covers:

1. Test cases that call bind (c) procedures written in C (callee) from Fortran (caller) with acceptable actual and dummy arguments of assumed-length character types (except complicated expressions in dummy argument bound or extent specification).

2. Test cases that call bind (c) procedures written in Fortran (callee) from Fortran (caller) or C (caller) with acceptable actual and dummy arguments of assumed-length character types (except complicated expressions in dummy argument bound or extent specification).


Applicable Architecture and

This feature has no dependency on any architecture/OS and the test cases should run on AIX, Linux (Big endian and little endian) and BG.

Objectives of the Test

Please see the test strategy section.


Test Strategy


Functional Tests:

Functional test cases contain scenarios that pass Fortran objects to C functions and vice versa using interfaces with the assumed-length dummy argument whenver needed. We test the following requirements:

  1. The length of an assumed-length object is assumed from its effective argument.

  2. When the dummy argument is an array, the lower bounds of each dimension of the CFI descriptor is 0 in C callee and 1 in Fortran callee.
  3. The content/properties of the character string of the assumed-length entity is expected to show correct behavior before, after and inside the BIND(C) routine.
  4. When a C function is invoked with an actual argument that corresponds to an assumed-length dummy argument in a Fortran interface, for that C function, the corresponding formal parameter is the address of a descriptor of type CFI_cdesc_t.
  5. The attribute member of a C descriptor for a non-allocatable and non-pointer assumed-length is CFI_attribute_other. The attribute of an assumed-length character array cannot be changed.
  6. An assumed-length character type dummy argument can have OPTIONAL attribute. If such an optional argument is not present, the corresponding formal parameter of the C function should be NULL.
  7. Since the dummy argument is always non-pointer, non-allocatable for this feature, any CFI function that may modify the CFI descriptor is non-conforming. Therefore, CFI functions: CFI_address, CFI_is_contiguous are okay. For CFI_section, CFI_select_part, the dummy argument CFI descriptor can only appear as the source. CFI_setpointer is also okay if the dummy argument CFI descriptor appears as the source (i.e. not the result). CFI_allocate, CFI_deallocate, CFI_establish, CFI_setpointer are not allowed.
  8. If an object is passed to a Fortran procedure as a non-allocatable non-pointer assumed-length dummy argument, its lifetime shall not end before the return from the procedure call. (Nested calls need to be used to test)
  9. The above mentioned rules should hold for all scopes (main program, module, sub-modules, internal/external subroutines and other constructs)
  10. Assumed-length dummy argument with INTENT is acceptable.
  11. In casese when the dummy argument is an explicit shape array or assumed-size array, if the actual argument is not contiguous, the Fortran caller is responsible to create a contiguous copy of the actual argument and pass the copy. In other words, the dummy argument in those cases will always be associated with a contiguous storage irrespective of the contiguity of the actual argument.
  12. The callee function should check the consistency of the dummy argument properties: base_addr, type, elem_len, content, rank, memory and dimension values, CFI function values (CFI_address, CFI_is_contiguous).
  13. Test of different calling sequences between Fortran and C code:
    - C-to-Fortran
    - C-to-Fortran-to-C
    - C-to-Fortran-to-Fortran (non-bindc)
    - Fortran-to-C
    - Fortran-to-C-to-Fortran
    - Fortran-to-Fortran
    - Fortran-to-Fortran-to-C
    - Fortran-to-Fortran-to-Fortran (non-bindc)

Diagnostic Tests:

1. Constraints mentioned in this I0 (dummy argument needs to be nonallocatable, nonpointer)

2. Langlvl testing (i.e -qlanglvl=2008std, -qlanglvl=ts)

3. Type mismatch between dummy and actual argument

4. Test different CFI functions that cannot be used (CFI_allocate, CFI_deallocate)

5. Actual argument can be assumed_rank only if the the dummy is assumed_rank as well

6. Verify that the attribute "VALUE" is not accepted

Unit Tests:

Diagnostic Tests:

1. Test an assumed-length dummy argument with either allocatable or pointer attribute is not interoperable.
2. Test declaring an assumed-length dummy argument in a Fortran procedure with BINC(C) is not standard conforming with -qlanglvl=2008std, but OK with -qlanglvl=ts.
