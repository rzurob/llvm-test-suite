Feature Overview
Fortran TS 29113 enhanced the language interoperability between Fortran and C. One of the enhancement is to allow a nonallocatable, nonpointer dummy argument of type CHARACTER with assumed-length to be interoperable with a formal parameter of the prototype that is a pointer to CFI_cdesc_t. Note that before TS 29113, an entity of type CHARACTER is interoperable only if the length is 1.

Key Requirements

To implement this enhancement introduced in Fortran TS 29113.

Key Design Issues

1. What is the ABI of Fortran/C caller and Fortran/C callee.
2. what is the rank and bounds of an effective array argument in Fortran caller.
3. What is the rank and bounds of an array dummy argument in Fortran callee.
4. What if the actual argument has allocatable or pointer attribute when the assumed-length dummy argument is nonallocatable and nonpointer.
5. Who should handle the contiguity?

Key Design Decisions

1. Since a nonallocatable, nonpointer dummy argument of type CHARACTER with assumed-length corresponds to a formal parameter of the prototype that is a pointer to CFI_cdesc_t (C descriptor), Fortran caller needs to convert the actual argument to a C descriptor effective argument. Fortran callee also needs to accept an incoming C descriptor dummy argument.
2. If the caller is Fortran, since the callee can be either Fortran or C, the rank and the bounds of a C descriptor effective argument should be set by the Fortran caller based on the declaration of the corresponding dummy argument in the interface block. Furthermore, the low bounds of the C descriptor effective argument should be set to 0 by the Fortran caller.
3. If the caller is Fortran, the C descriptor effective argument should be set properly. However, since the caller can also be C, the Fortran callee should set the bounds of the incoming C descriptor based on the declaration of the corresponding dummy argument. Further more, since the callee is written in Fortran, the low bounds should be adjusted to 1s if they are not explicitly specified.
4. Since the dummy argument must be nonallocatable and nonpointer, The actual argument C descriptor must not be changed. Fortran caller and callee need to set the CFI_attribute_other for the C descriptor effective argument or dummy argument respectively.
5. The contiguity is an issue only if the dummy argument is assumed-size or explicit-shape. In this case, Fortran caller or callee with handle the contiguity.

User Documentation

Overview

Fortran TS 29113 enhanced the language interoperability between Fortran and C. One of the enhancement is to allow a nonallocatable, nonpointer dummy argument of type CHARACTER with assumed-length to be interoperable with a formal parameter of the prototype that is a pointer to CFI_cdesc_t. Note that before TS 29113, an entity of type CHARACTER is interoperable only if the length is 1.

User Benefits

It is an improvement of the language interoperability. Users now can passing character strings with length greater than 1 between Fortran and C.

Public Interface

This feature is a functional enhancement for the language interoperability. There is not new syntax introduced.

Constraints

An assumed-length character type dummy argument of a BIND(C) routine must be nonallocatable and nonpointer.

Rules

1. If the caller is written in Fortran, users can pass any character constant, character variable or expression of type CHARACTER actual argument that corresponds to an assumed-length character dummy argument of a BIND(C) procedure no matter if the callee is written in Fortran or C. The C descriptor effective argument that is passed by Fortran caller has the rank and the extents based on the declaration of the corresponding dummy argument and zero low bounds. It also has the CFI_attribute_other flag set.

2. If the caller is written in C and the callee is written in Fortran, users need to pass a formal parameter of the prototype that is a pointer to CFI_cdesc_t corresponding to an assumed-length character dummy argument of a BIND(C) procedure.

3. If the callee is written in Fortran, the asummed-length dummy argument has the rank and bounds of what it is declared in the Fortran callee. The low bounds are 1s if they are not explicitly specified.

4. Since the assumed-length dummy argument must be nonallocatable and nonpointer, the incoming C descriptor effective argument that describes the rank and bounds of the actual argument must be intact after the call (This is not a issue if the caller is written in Fortran as the Fortran caller sets up the rank and bounds based on the dummy argument declaration in the interface block).

Language Standard Considerations

Fortran 2008 only allows CHARACTER(1) to be interoperable. TS 29113 relaxed the rule by allowing assumed-length character dummy argument to be interoperable. The compiler will flag a langlvl message for an assumed-length dummy argument in a BIND(C) procedure.

Messages

COMPONENT: 18
COMPONENT MESSAGE NO: 397
STATUS:
NAME: LANGLVL_ASSUMED_LEN_BINDC
SEVERITY: L
MESSAGE TEXT:
Procedure %1$s was declared with the BIND(C) attribute. Dummy argument %2$s of %1$s must not have assumed-length by the %3$s standard.

High-Level Implementation Design

Implementation Design Overview

1. If the caller is written in Fortran, the compiler will create a temporary C descriptor effective argument that has the the rank and extents of the corresponding dummy argument and zero low bounds wit the CFI_attribute_other flag set. It will also handle the contiguity on the caller side.
2. If the callee is written in Fortran, the compiler will create a temporary C descriptor dummy argument that holds the address of the incoming effective argument. It has memory type of mt_arg. The user declared dummy argument is changed to mt_auto. The mt_auto entity is used through out the function body. It describes the characteristics of the dummy argument of what is declared in the Fortran callee. Its low bounds are 1 if they are not explicitly specified and has the CFI_attribute_other flag set, so that the incoming C descriptor effective argument is intact. The Fortran callee will handle the contiguity. Because of the above implementation design, the debug code needs to generate a TEST for the mt_auto to represent the dummy argument rather than the temporary C descriptor.
