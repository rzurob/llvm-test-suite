Feature Overview


C-Interop TS29113 removed some restrictions on ISO_C_BINDING module procedures.
  • C_LOC: The argument is no longer required to be scalar if it is type and type parameters are not interoperable (F2008 requirement). The pointer is no longer required to be scalar (F2003 requirement)
  • C_FUNLOC: The argument is no longer required to be interoperable procedure.

Key Requirements


To implement the change and make sure to issue different language level messages.

Key Design Issues


None.

Key Design Decisions


None.

Testing Strategy


This is a unit test only feature. The unit test will cover
  • passing noninteroperable array actual argument to C_LOC will not fail the compilation. The same argument will trigger the langlvl message for F2008 standard and before.
  • passing a pointer scalar argument to C_LOC will not fail the compilation. The same argument will trigger the langlvl message for F2003 only.
  • passing noninteroperable procedure to C_FUNLOC will not fail the compilation. The same argument will trigger the langlvl message for F2008 standard and before.

Language Standard Considerations
F2008 standard requires the argument to C_LOC whose type and type parameters are not interoperable to be scalar nonpolymorphic variable with no length type parameters.
TS29113 removed the scalar requirement from the above restriction.

F2008 standard requires the argument to C_FUNLOC to be an interoperable procedure.
TS29113 removed the above restriction.

F2003 standard requires the pointer argument to C_LOC must be scalar.
F2008 removed the above restriction.

Messages
The following language level message have been approved by the XLF Message committee.
  • Argument %1$s to the intrinsic procedure C_LOC has the %2$S attribute. %1$s must be a scalar by the %3$s standard.
  • Argument %1$s to the intrinsic procedure C_LOC does not have interoperable type and type parameters. %1$s must be a scalar nonpolymprphic variable with no length type parameters by the %2$s standard.
  • Argument %1$s to the intrinsic procedure C_FUNLOC must have the %2$S attribute by the %3$s standard.

Rules for file name generation on Z/OS

 

  1. There are 4 types of invocation: JCL, TSO, c89 utility and xlc utility
  2. JCL invocation is processed separate from all other invocation types. A default destination name is generated based on the source file name. Output file names are then generated based on this default destination name. For example:
    • If I compile a source file //'TSCTEST.PDS.C(FOO)' the generated name of the DBRM file would be //'MILOS.PDS.C.DBRM(FOO)'
    • If I compile a source file //'TSCTEST.SEQ.C' the generated name of the DBRM file would be //'MILOS.SEQ.C.DBRM'
    • If I compile a source file /home/milos/foo.c the generated DBRM file would be ./foo.dbrm
  3. TSO invocation relies on REXX execs which specify a default destination pattern "//~.DEST". The default destination pattern is passed to the compiler as an argument. This pattern causes the compiler to generate output file names using the same logic as in the JCL case.
  4. The xlc utility always generates a default destination name based on the source file name (this can not be changed by the user). The default destination name is passed to the compiler as an argument. The compiler uses the default destination name to generate output file names in a similar fashion as in the JCL case, but with slightly different results. For example:
    • If I compile a source file //'TSCTEST.PDS.C(FOO)' the generated name of the DBRM file would be //'MILOS.PDS.DBRM(FOO)'
    • If I compile a source file //'TSCTEST.SEQ.C' the generated name of the DBRM file would be //'MILOS.SEQ.DBRM'
    • If I compile a z/OS UNIX file /home/milos/foo.c the generated DBRM file would be ./foo.dbrm
  5. Like xlc, the c89 utility always generates a default destination name based on the source file name. By default the behavior is the same as in the xlc case. However, this can be changed by setting _OSUFFIX_HOSTQUAL and _OSUFFIX_HOSTRULE to 0 which yields the same behavior as in the JCL case.}%
