Feature Overview

This feature will implement complex-part-designators. A complex-part-designator allows the user to directly access the real or imaginary part of a complex number through the use of %RE and %IM part-designators, respectively.

Key Requirements

The functionality for complex-part-designators is required for continued compliance with the Fortran 2008 standard.

Testing Strategy

Unit tests will include diagnostic testing for the constraints stated below. Functional testing will be written to ensure that complex-part-designators will behave functionally as a REAL object designator.

Further testing:
  1. Testing should include all the syntax rules R615, R601, R618 and C624 ref: Fortran 2008 Standard.
  2. Different language standards should be tested.
  3. Beside checking the precision of results, the exact value designated should also be checked, by comparing the designated value with the returned value by other Intrinsics e.g: REAL and AIMAG etc.
  4. Test array designator as a complex-part-designator ref: Section 6.5.4 of the Fortran 2008 Standard.
The details and some examples are provided as samples in Section Testing Considerations.

User Documentation

Overview

A complex-part-designator allows the user to directly access the real or imaginary part of a complex number through the use of %RE and %IM part-designators, respectively.

Instead of constructing new complex numbers when changing either the real or imaginary part of a complex, the user can access the specific parts directly. Also, these can be used instead of the intrinsics REAL() and IMAG().

Public Interface

New syntax is available to the user to access the real and imaginary parts of a complex number.

Constraints

  1. A complex-part-designator must only appear on a designator of complex type.
  2. Exactly one part-ref shall have nonzero rank. Either the final part-ref shall have a section-subscript-list with nonzero rank,
    another part-ref shall have nonzero rank, or the complex-part-designator shall be an array.
  3. A complex part designator cannot appear within an EQUIVALENCE statement.

Rules

  1. Complex-part-designators must follow rules specified for real data types.
  2. The type of a complex part designator is real and its kind and shape are those of the designator
  3. Complex part designators can appear as the (lhs) in the assignment statement.
  4. An array designator with two or more elements and a complex-part-designator is not contiguous.

Language Standard Considerations

  1. The complex-part-designator is a Fortran 2008 Standard feature. The use of a complex-part-designator when compiling with -qlanglvl=2003std or below will result in a language level message.

Messages

Messages will be proposed and approved by the XLF Messages committee.

Three new messages will be proposed to the XLF Messages committee.

Performance Considerations

The use of complex-part-designators will not have an impact on performance.

complex :: cmplx

subroutine assignReal(in,out)
real, intent(in) :: in
real, intent(out) :: out
out = in
end


complex, target :: ctar
real, pointer :: realp
ctar = (0.1,0.2)
realp => ctar%re
print *, realp ! 0.1
realp => ctar%im
print *, realp ! 0.2
end

complex, pointer :: ptrc
real, target :: tarr
allocate ( allocc%im ) ! Invalid
ptrc%im => tarr ! Invalid
end

Testing Considerations

The following two types of testing will be considered:

Diagnostic:

  • Test with different syntax and different types of designators, like integer, real, character and logical etc.
  • Test type and kind of the complex part designator.
  • Test and check all the warning messages as approved by the XLF messages committee.
  • Test the use in equivalence statement as equivalence objects.
  • Some examples for diagnostic testing mentioned above:
 
      complex(8) c1, c2
     real r
 
      r%IM                                 ! Type error
      r = (1.0,0.0)%RE                     ! Syntax error
      c1%RE => 1.5                         ! Syntax error
      c1%RE = 1.5
      c1%IM = 0.0
      c2%RE = c1%IM
      c2%IM = c1%RE
 
      print *, c1                          ! Should print (1.5,0.0)
      print *, c2                          ! Should print (0.0,1.5)
      print *, KIND(c1%RE)
      print *, KIND(c2%IM)                 ! Should print 8 in both cases
      
      r%RE                                 ! Type error
      EQUIVALENCE (r, c1%RE), (r, c1%IM)   ! Invalid
      end

Functional:

  • Test the use as an actual argument, both scalar and array, in a module, a function, a subroutine and an elemental function
  • Set (sample given above) and test the real and imaginary parts
  • Test the use in different expressions, initialization, constant and assignment etc
  • Test the use in allocate and pointer assignment statements (for examples see section Samples in User Documentation)
  • Check if an array section as a complex part designator is contiguous or not
e.g:
 
            complex, dimension(7) :: c
            is_contiguous(c%RE)        ! Returns false
            end
  • Consider the following when verifying the designated value:
  • Compare the results in different expressions
e.g:
 
            complex(8) c1
            c1%RE = 1.5
            sin(c1%RE) == sin(1.5)     ! Should be exactly the same
            end
  • Test the value designated using the intrinsic functions REAL and IMAG etc
e.g:
 
            complex :: c
            c = (1.0,2.0)
            c%RE == real(c)
            c%IM == imag(c)
            end
  • Compare against the already known value

High-Level Implementation Design

Implementation Design Overview

The front end will parse complex-part-designators. W-Code++ will be emitted which describes the layout of complex data types with complex-part-designators. The W-Code++ will contain an ildb-class SYM for each complex-part-designator, as a member of the owning complex ild-class SYM. The ildb-class SYMs, for the complex-part-designators, will have the type dt=float, whereas the complex will have dt=complex. ASTI will consume and emit similar W-Code to be consume by downstream components.

An ildb-class SYM for a complex-part-designator, such as C%RE and C%IM, will be aliased to its owner, the ild-class SYM for the containing complex object, C. If complex A is aliased with complex B, then A%RE will be aliased with B%RE and A%IM will be aliased with B%IM

n addition to aliasing the %RE and %IM parts with their containing complex object:

- The %RE and %IM parts must be aliased with the shadow for REAL of their precision, and the shadow for COMPLEX of their precision.

- The containing complex object must be aliased with the shadow for COMPLEX of its precision, and with the shadow for REAL of its precision.

- The shadows for COMPLEX and REAL of the same precision must be aliased.

These are the same rules as if COMPLEX were considered a derived type (C struct) containing the two REALs.

Similar rules apply for vector complex types.

Inter-Component Communication/Interfaces

The W-Code stream will contain ildb-class SYMs for each complex-part-designator as members of the owning complex ild-class SYM. For example:

SYM id_no=2 len=8 owner=1{"_main"} sym_flags=( ) flags=( )
dt=complex mt=auto align=8 seg_no=0 nlen=1 name=0x46{"c"};
SYM{b} id_no=6 len=4 owner=2{"c"} offset=0 alias_no=0 flags=( )
dt=float linkage=unknown nlen=4 name=0x82{"c%re"};
SYM{b} id_no=8 len=4 owner=2{"c"} offset=4 alias_no=0 flags=( )
dt=float linkage=unknown nlen=4 name=0x88{"c%im"};
