Overview

%Note{A brief description of the feature, its application(s) and limitation(s).

Feature syntax may also be mentioned here. Since it is in the I0 document, it is optional.}%

The Fortran 2008 language standard introduced the DO CONCURRENT construct. This construct expects do loop iterations to have no interdependencies, therefore iterations can be completed concurrently if specified. Note that in XL Fortran V15.1.3, DO CONCURRENT loops execute sequentially only. In future releases, DO CONCURRENT will have a parallel implementation.

New syntax:
do concurrent([ integer-type-spec :: ] concurrent-triplet -list [, scalar-mask-expr ])
  [do concurrent block body ]
end do
  • integer-type-spec is an integer type specification
  • concurrent-triplet is index-name = concurrent-limit : concurrent-limit [ : concurrent-step ]
  • concurrent-limit is a scalar-int-expr concurrent-step is a scalar-int-expr
  • mask-expr is a logical-expr

Examples:
logical :: logarr(10) = .true., result(10) = .false.
logarr(5) = .false.
do concurrent(integer :: i = 1:5, integer :: j = 1:10, logarr(j) .eqv. .true.)
  result(i) = logarr(j)
end do

You can have any number of iterators and any number of scalar-mask-expr's you would like. When specifying multiple scalar-mask-expr's, one must use logical operators to combine them. Below is an example:
integer :: i, j, i_arr(5), i_arr2(5), i_res(5,5) = 0
i_arr = (/ 11, 22, 33, 44, 55 /)
i_arr2 = (/ 10, 20, 30, 40, 50 /)
do concurrent (i = 1:5, j = 1:5, i_arr2(i) > 20 .and. i_arr(j) < 50)
  i_res(i,j) = i_arr(j)*i
end do

Language constraints:
  1. Any procedure referenced in the scalar-mask-expr, including one referenced by a defined operation, shall be a pure procedure.
  2. The index-name shall be a named scalar variable of type integer.
  3. A concurrent-limit or concurrent-step in a concurrent-control shall not contain a reference to any index-name in the concurrent-control-list in which it appears.
  4. If the do-stmt of a do-construct specifies a do-construct-name, the corresponding end-do shall be an end-do-stmt specifying the same do-construct-name. If the do-stmt of a do-construct does not specify a do-construct-name, the corresponding end-do shall not specify a do-construct-name.
  5. A cycle-stmt shall not appear within a DO CONCURRENT construct if it belongs to an outer construct.
    • cycle-stmt: cycle-stmt is CYCLE [ do-construct-name ]
  6. A RETURN statement shall not appear within a DO CONCURRENT construct.
  7. A branch within a DO CONCURRENT construct shall not have a branch target that is outside the construct.
  8. A reference to a nonpure procedure shall not appear within a DO CONCURRENT construct.
  9. A reference to the procedure IEEE_GET_FLAG, IEEE_SET_HALTING_MODE, or IEEE_GET_HALTING_MODE from the intrinsic module IEEE_EXCEPTIONS, shall not appear within a DO CONCURRENT construct.
  10. An exit-stmt shall not appear within a CRITICAL or DO CONCURRENT construct if it belongs to that construct or an outer construct.
  11. If the do-stmt is a nonlabel-do-stmt, the corresponding end-do shall be an end-do-stmt.
  12. If the do-stmt is a label-do-stmt, the corresponding end-do shall be identified with the same label.

Test Strategy

%Note{This is the most important section of the document.

Detail the test methodology and coverage for the feature to be tested.

Test strategy is not to identify all possible defects under all conditions with the feature. In most cases, this is unrealistic.
It is to identify and report defects with the feature under specific conditions likely to be used by the customer.
As a tester, you are the first customer of the feature. So you, the tester, need to understand how a certain feature would be used in the real world and come up with a Test strategy to validate those use cases.

Use cases to be validated should be documented here.

Provide examples if they are needed to explain the test method or coverage.

If applicable, organize the coverage into diagnostic and functional sections.

Diagnostic test cases verify error handling of the feature and the corresponding messages that are produced. This would also include verifying that valid syntax does not produce any unexpected messages.
This type of testing usually fall under the umbrella of unit tests. The developer of the feature is expected to write fully automated unit tests, but this can be negotiated if it is unfeasible for the developer to do so.

Test cases that verify values are referred to as functional test cases.

For large features, there may not be resource/time to completely test the feature. Certain parts of the feature will be more important than others.
Prioritize the test coverage and clearly document what will be covered in testing and what will be omitted. Risk assessment on the parts omitted should also be documented. This needs to be reviewed and agreed to by the requirements owner(s).

If there is already coverage in existing test suites, explicitly document it here.

Often, a feature by itself will work fine, but when combined with existing functionality or other new features, it may not work correctly. So ensure this type of testing is covered in the test coverage. This implies that other new features being developed for the release should also be understood.

Backward compatibility, portability, and any other migration Issues should be considered. One of the main complaints from customers is migrating from release to release (or from one platform to another).

Document any dependencies or open issues with the feature, so these are not over-looked.}%

Testing will begin by focusing on the following main areas.

SECTION ONE:
Allocatables,pointers; derived types; and scalars, arrays, subobjects:
  • inside DO CONCURRENT loops (in scalar-mask-expr and in loop body)
  • in DO CONCURRENT loops inside blocks
  • in blocks with DO CONCURRENT loops inside them
  • in blocks with DO CONCURRENT loops inside them with blocks inside them
  • in DO CONCURRENT loops nested in other loop structures (DO loops for example)
  • in other loop structures (DO, FORALL, indirect array loops, etc.) nested inside DO CONCURRENT loops
declared in:
  • the host program
  • a module
  • a submodule
  • a procedure
  • a block

Examples of subobjects covered:

  • An array element
  • An array section
  • A character substring
  • A complex part
  • A structure component

There will be tests covering the scope in which the DO CONCURRENT appears including:

  • the main program
  • a module procedure
  • an internal procedure
  • an external procedure
Also, we will minimally test indirect array access in scalar-mask-expr's and DO CONCURRENT loop bodies.

The types we will test will include integers, reals, complex, logicals and characters of different kinds, as well as double precision and user-defined types.

SECTION TWO:
Scenarios will also be written covering the criteria in the above list using functions and subroutines inside the DO CONCURRENT loops (inside the loop body and inside the scalar-mask-expr). Any procedure in the scalar-mask-expr must be a pure procedure.  At least one scenario will be added to cover the interaction of variables used in DO CONCURRENT and all applicable attributes. Both user-defined and intrinsic procedures will be tested, including elemental procedures.

SECTION THREE:
We will also be analyzing sections 16.6.3, 16.6.5 and 16.6.7 (see below) of the Fortran language standard for applicable coverage. Section 16.6.3 lists situations where variables are initially defined. Section 16.6.5 lists situations where variables become defined. Section 16.6.7 outlines variable definition context. In addition to the coverage outlined in sections one and two, we will scan these sections in the standard for other applicable features we can test with the DO CONCURRENT construct.

SECTION FOUR:
Since we do not offer a parallel implementation of DO CONCURRENT this release, OMP doesn't support F2008 at this time and !$OMP DO does not accept DO CONCURRENT as an acceptable DO loop, we will be doing acceptance tests for some OMP directives in case the customer uses them.

Note that !$OMP PARALLEL will not parallelize the loop iterations, it will execute the entire loop once for each thread.

We will be using existing test cases in our test suite and converting them to use DO CONCURRENT loops to make our acceptance test cases.

SECTION FIVE:
Diagnostic test cases will be created to cover illegal use cases where error messages are expected including the constraints listed in the overview of this document. From the constraints listed in the Overview of this document, each constraint will have at least one diagnostic test case written for it (constraints 4, 11 and 12 can be grouped together in the same test case).
