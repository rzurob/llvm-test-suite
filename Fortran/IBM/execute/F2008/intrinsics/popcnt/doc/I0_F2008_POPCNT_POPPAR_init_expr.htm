Access Change History

 Access Change History

Feature Overvie

POPCNT and POPPAR are new intrinsics added in the Fortran 2008 standard. XL Fortran already supports these two intrinsics as IBM extensions. Currently the functions are evaluated during run time, and are not allowed as constant expressions. To fully comply with the F2008 standard, this feature allows the two functions to be used in constant expressions.

Key Requirements

The key requirement is to allow the functions in constant expressions. The compiler no longer issues an error message when then functions are used as constant expressions.

N/A

N/A

Testing Strategy

The POPCNT and POPPAR functions will be tested in constant expressions. Langlvl messages will also be tested.

User Documentation

Overview

POPCNT and POPPAR functions are allowed in constant expressions.

User Benefits

The feature extends the functionality of the POPCNT and POPPAR functions, allowing them to be used in constant expressions. This brings these functions into compliance with the Fortran 2008 standard.

Public Interface

A function call to POPPAR and POPCNT can be used where a constant expression is necessary in an initialization expression or specification expression.

e.g: integer::i = poppar(5)

real, dimension(5, popcnt(5)) :: arr

Language Standard Considerations

The functions are IBM language extensions, now they are part of the F2008 standard. If langlvl lower than 2008 is specified, the compiler will continue to issue a langlvl message.
The F2008 standard only support argument of integer type. XL Fortran supports real, byte and logical in addition to integer. there will be langlvl checking on the argument type. When F2008 langlvl is specified, the compiler will issue a langlvl message when the argument is not of integer type.

Messages

Existing langlvl messages will be used for this feature.

The error message issued when the POPPAR and POPCNT functions are used as constant in initialization expressions is removed.

Migration/Co-Existence/Binary-Compatibility Considerations

Information Development Impacts

Some documentations needs updating, the language reference in particular. POPCNT and POPPAR with argument type of integer are no longer IBM extensions in F2008. The support for other argument types other than integer types remain as IBM extension.

The following ID impact source is from ID:

  • Fortran Language Reference
    • Chapter 14 Intrinsic procedures
      • Classes of intrinsic procedures
        • Mark POPCNT and POPPAR as F2008.
      • POPCNT(I) (IBM extension)
      • POPPAR(I) (IBM extension)
        • Modify the titiles of these two topics to POPCNT(I) (Fortran 2008) and POPPAR(I) (Fortran 2008).
        • In the "Argument type and attributes" section, mark all the types except integer as "IBM extension".
        • In the "Result type and attributes" section, add the description that POPCNT and POPPAR can be used in constant expressions.
        • In the "Examples" section, provide examples that these two functions are used in constant expressions.

Testing Considerations

Test cases need to be updates if they test error messages regarding the usage of POPCNT and POPPAR function in constant expressions.

High-Level Implementation Design

Implementation Design Overview

For the compile time evaluation of the POPCNT and POPPAR, the frontend calls a runtime routine to evaluates the intrinsic. The runtime routine is a wrapper that calls POPCNT or POPPAR TOBEY BIF.

  1. The "allowed in initialization" and "allowed in specification expression" attribute flags in math_tab.def should be set to Fortran 2008 Standard and xlf (IBM extension). The functions should also be marked as conforming to the Fortran 2008 standard.
  2. invok_if.cpp file will add the call to POPCNT and POPPAR runtime routine wrapper.
  3. POPCNT and POPPAR wrapper will be created in the Fortran run time, the wrapper will call the TOBEY builtin POPCNT and POPPAR to perform the function evaluation.

Inter-Component Communication/Interfaces

The FE calls the POPPAR and POPCNT TOBEY builtins to evaluate functions during compile time.
