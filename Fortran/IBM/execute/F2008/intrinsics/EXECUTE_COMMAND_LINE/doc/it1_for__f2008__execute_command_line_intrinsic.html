Overview

Fortran 2008 adds the EXECUTE_COMMAND_LINE intrinsic which is similar to XLF's SYSTEM intrinsic (IBM extension). EXECUTE_COMMAND_LINE will allow the user to execute a command line from the FORTRAN program including starting another program. The F2008 standard specifies that the synchronous execution must be supported whereas the asynchronous execution might be. Currently, the XLF compiler will only support the synchronous execution.

EXECUTE_COMMAND_LINE is specified in section 13.7.57 of the Fortran 2008 standard. The syntax is as follows:

EXECUTE_COMMAND_LINE(COMMAND [, WAIT, EXITSTAT, CMDSTAT, CMDMSG ])


  • COMMAND shall be a default character scalar. Its value is the command line to be executed. The interpretation is processor dependent.

  • WAIT (optional) shall be a default logical scalar. If WAIT is present with the value false, and the processor supports asynchronous execution of the command, the command is executed asynchronously; otherwise it is executed synchronously.

  • EXITSTAT (optional) shall be a default integer scalar. If the command is executed synchronously, it is assigned the value of the processor-dependent exit status. Otherwise, the value of EXITSTAT is unchanged.

  • CMDSTAT (optional) shall be a default integer scalar. It is assigned the value -1 if the processor does not support command line execution, a processor-dependent positive value if an error condition occurs, or the value -2 if no error condition occurs but WAIT is present with the value false and the processor does not support asynchronous execution. Otherwise it is assigned the value 0.

  • CMDMSG (optional) shall be a default character scalar. If an error condition occurs, it is assigned a processor-dependent explanatory message. Otherwise, it is unchanged.

The standard introduces one constraint on the use of EXECUTE_COMMAND_LINE:


  • If a condition occurs that would assign a nonzero value to CMDSTAT but the CMDSTAT variable is not present, error termination is initiated.

EXECUTE_COMMAND_LINE has a similar behavior as SYSTEM (IBM extention). SYSTEM has been previously tested in particular for HPF (High performance Fortran). See the following bucket for some examples:

/tstdev/hpf_intrin_sub/system/

Note that the EXECUTE_COMMAND_LINE test cases will not be run on BG (BG does not support command line execution. So if execute_command_line is run on BG, the command will not run and cmdstat will always be set to -1)


Test Strategy

The feature needs both Diagnostic and Functional testing. The diagnostic test will ensure that the intrinsic subroutine EXECUTE_COMMAND_LINE will behave according to the specification of the F2008 standard and verify that messages appear where appropriate.

Diagnostic testing:

  1. Test that the compilation fails if COMMAND is omitted
  2. Test that the compilation fails if COMMAND is specified more than once
  3. Test that the tc compiles successfully using the argument keyword COMMAND
  4. Test that the compiler properly diagnose mismatching types/ranks of the arguments

Fuctional testing:

The test strategy for the functional testing is to verify the correct execution of the program or the command line started by the EXECUTE_COMMAND_LINE intrinsic. Different Unix/Linux commands will be used (ie. echo, mv, date,...). Different verification files might need to be generated depending on the platform. Running a different Fortran program will also be tested using EXECUTE_COMMAND_LINE.

  1. Test whether the user specified command can be executed correctly and the correctness of the results need to be checked
    • run shell commands ie.: uname/echo/mv/cat
    • run Fortran executable (created in the scenario file) and verify data
  2. Test that no error message is emitted when WAIT, EXITSTAT, CMDSTAT or CMDMSG are omitted or present and that CMDSTAT is not set to -1
  3. Test that CMDSTAT =-2 when WAIT is set to .False. and no error condition occurs
  4. If a condition occurs that would assign a nonzero value to CMDSTAT but the CMDSTAT variable is not present, error termination is initiated. We should test if the program will be terminated:
    • CMDSTAT and CMDMSG not present and an error condition occurs => error termination of the program
    • CMDSTAT not present but CMDMSG present and an error condition occurs => error termination of the program
  5. Whether EXITSTAT, CMDSTAT or CMDMSG argument can be set correctly by program according to the F2008 standard if they are specified:
    • CMDSTAT present but CMDMSG not present and an error condition occurs => the program runs successfully (normal termination at the end of the program)
    • CMDSTAT and CMDMSG both present and an error condition occurs => CMDSTAT and CMDMSG values change
    • CMDSTAT and CMDMSG both present and no error condition => CMDMSG value unchanged and the CMDSTAT is set to 0
