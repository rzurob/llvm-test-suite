Feature Overview

Fortran 2008 introduces LCOBOUND and UCOBOUND intrinsics, which return the lower and upper cobounds of a coarray object respectively.

Key Requirements


These intrinsics are required for conformance with the F2008 standard in supporting Coarray features.

Key Design Issues


To support these intrinsics, the compiler needs to recognize these intrinsics, perform semantic checking, and generate code to calculate the result. The cobounds of a coarray can be one of the following:
  • constant expression (eg. static coarrays)
  • specification expression (ie. coarray dummy arguments or local coarrays of a subprogram or a BLOCK construct)
  • deferred (ie. allocatable coarray)
In the V14.1 release, we don't need to support allocatable coarrays, however the design must be flexible to allow future support for the allocatable coarrays.

We also need to consider cases where these intrinsics are used in constant expressions or specification expressions.

Key Design Decisions


To recognize LCOBOUND and UCOBOUND as intrinsic functions, we will add corresponding entries in the math table in FE and specify the properties of the arguments so that basic semantic checking can be performed without much effort. In cases where these intrinsics are used in constant expressions, FE needs to inline the results. In all other cases, FE will generate a BIF that will be consumed by ASTI to generate the runtime code for calculating the result. ASTI uses the cobounds information as well as the information about the number of images available in W-Code++ to calculate the lower and upper cobounds for each codimension.

An issue has been raised regarding programs complied without -qcaf that contain calls to these intrinsics. The concern is to be able to diagnose this situation. The agreement is that the existing constraints (see the Constraints section) are sufficient to prevent miscompilation in such cases. The COARRAY argument of these intrinsics need to be coarray. This constraint is checked by the compiler and if violated causes the compilation to halt with a sever message. On the other hand, to be able to pass a coarray to these intrinsics, the coarray must have been declared in the current compilation unit, or it must have come from a module. If the declaration is part of the current compilation unit, then -qcaf must have been specified, otherwise the declaration would be in error. If the coarray object comes from a module, then that module must have been compiled with -qcaf, as we would not allow modules with conflicting -qcaf settings to coexist in an application (we will flag the USE statement).

Testing Strategy

  • Unit testing will cover diagnostic as well as functional test cases for LCOBOUND and UCOBOUND as described in the Testing Consideration section.
  • Test team will provide additional coverage as they see fit. (perhaps to include the interaction of these intrinsics with other language features).
Note: Implementation and testing will only cover non-allocatable coarrays at this point. Test team should consider these intrinsics when we do the allocatable coarray feature in future releases.
User Documentation

Overview


The cobounds of a coarray define the range of allowable coindexes in each codimension. Except for the upper cobound of the last codimension, cobounds of a coarray are determined by the values specified in square brackets in the declaration or allocation of that coarray. The upper cobound of the last codimension is the last cosubscript for COARRAY that selects the image with index NUM_IMAGES().

The LCOBOUND intrinsic allows users to inquire about the lower cobound of each codimension in a coarray, or the lower cobound of a specific codimension.

The UCOBOUND intrinsic allows users to inquire about the upper cobound of each codimension in a coarray, or the upper cobound of a specific codimension.

Note: The corank of a coarray is equal to the number of its codimensions. F2008 standard specifies that the sum of rank and the corank of an entity must not exceed 15. XL Fortran allows the sum of rank and the corank to go up to 20.

User Benefits

This feature allows users to inquire about the cobounds of coarray objects in their application.

Public Interface


LCOBOUND(COARRAY, DIM, KIND)

Purpose

Returns the lower cobound of each codimension in a coarray, or the lower cobound of a specified codimension.

Class

Inquiry function

Argument type and attributes

COARRAY

is the coarray whose lower cobounds are being inquired. It can be an array or a scalar of any type. If it is an allocatable coarray, it must be allocated.

DIM (optional)

is an integer scalar in the range 1 <= DIM <= corank(COARRAY). The corresponding actual argument must not be an optional dummy argument.

KIND (optional)

must be a scalar integer initialization expression.

Result type and attributes

  • It is of type integer.
  • If KIND is present, the KIND type parameter is that specified by the value of KIND; otherwise, the KIND type parameter is that of default integer type.
  • If DIM is present, the result is a scalar. If DIM is not present, the result is a one-dimensional array with one element for each codimension in COARRAY.

Result value

LCOBOUND(COARRAY) returns an array where each element of the array contains the value of the lower cobound of the corresponding codimension.

LCOBOUND(COARRAY, DIM) is equal to the lower cobound for cosubscript DIM of COARRAY.


UCOBOUND(COARRAY, DIM, KIND)

Purpose

Returns the upper cobound of each codimension in a coarray, or the upper cobound of a specified codimension.

Class

Inquiry function

Argument type and attributes

COARRAY

is the coarray whose upper cobounds are being inquired. It can be an array or a scalar of any type. If it is an allocatable coarray, it must be allocated.

DIM (optional)

is an integer scalar in the range 1 <= DIM <= corank(COARRAY). The corresponding actual argument must not be an optional dummy argument.

KIND (optional)

must be a scalar integer initialization expression.

Result type and attributes

  • It is of type integer.
  • If KIND is present, the KIND type parameter is that specified by the value of KIND; otherwise, the KIND type parameter is that of default integer type.
  • If DIM is present, the result is a scalar. If DIM is not present, the result is a one-dimensional array with one element for each codimension in COARRAY.

Result value

UCOBOUND(COARRAY) returns an array where each element of the array contains the value of the upper cobound of the corresponding codimension.

UCOBOUND(COARRAY, DIM) is equal to the upper cobound for cosubscript DIM of COARRAY.

The upper cobound of the last codimension is the last cosubscript for COARRAY that selects the image with index NUM_IMAGES().

Constraints

Compiler will check for the following constraints and issue messages when they are violated:

  1. The actual argument associated with COARRAY must be a coarray.
  2. The actual argument associated with DIM must be a scalar integer.
  3. The actual argument associated with DIM must have a value in the range 1 <=DIM <=corank(COARRAY). For performance reasons, compiler will only detect this when the value of DIM is known at compile-time.
  4. The actual argument associated with KIND must be a scalar integer that is known at compile-time.
  5. The value of the KIND argument must be a valid value for KIND type parameter of an integer.
  6. Argument keywords must match the names described in the public interface section.
  7. The actual argument associated with DIM must not be a an optional dummy argument.
  8. If an LCOBOUND or UCOBOUND appears in a constant expression, its result must be compile-time known.

Rules

  • If the actual argument associated with COARRAY is an allocatable coarray, it must be allocated before calling LCOBOUND or UCOBOUND.
  • Runtime values of DIM must be in the range 1 <= DIM <=corank(COARRAY). If this condition is violated, incorrect results will be produced.

Language Standard Considerations

These intrinsics are part of Fortran 2008 standard. They will be flagged when -qlanglvl is set to a level that does not include 2008 features.

Messages

We have existing messages for all (except one) of the cases mentioned in the constraints section. Constraint #3 is being discussed in the message committee.

       integer :: ucobds(3)
       
       ucobds = ucobound(coarr, kind=4)
       
       if (num_images() .eq. 30) then
          print *, ucobds  ! should print 3, 7, 2
       end if
       
       end

       call sub(c, 2)
       
       contains
       subroutine sub(coarr, n)
         integer :: n
         integer :: coarr[n:*]
         print *, lcobound(coarr,1) ! should print 2
       end subroutine
 
       end

The following cases need to be considered for testing:

Diagnostic:

  • All cases mentioned in the constraints section.

Functional:

  • Scalar coarrays as well as array coarrays
  • Coarrays with corank 1
  • Coarrays with corank > 1
  • Coarray declared with default lower cobound.
  • Static coarrays (constant expression cobounds)
  • Explicit-coshape coarray declared as dummy argument with constant/non-constant cobound expressions.
  • Explicit-coshape coarrays declared as a local coarray of a subprogram or a BLOCK construct with constant/non-constant cobounds expressions.
  • DIM is present/not present
  • KIND is present/not present
  • DIM and KIND are both present/not present
  • COARRAY, DIM, KIND argument keywords used in different orders
  • UCOBOUND of the last codimension with various number of images (specifically cases where the number of images is/is not large enough to span the entire extent of previous codimensions).
  • Interaction with -qintsize when KIND is present/not present
  • Specification expressions
  • Constant expressions

High-Level Implementation Design

Implementation Design Overview

In the case of constant expressions, FE should inline the result. In all other cases, FE generates BIFs that will be consumed by ASTI to generate code to calculate the result.

Inter-Component Communication/Interfaces

FE will generate the __lcobound_with_kind() and __ucobound_with_kind() BIFs. These BIFs always have 3 parameters. In the case of missing KIND, FE will provide the default integer size value. In the case of missing DIM, FE will generate a call to __supply_opt_arg() in place of the absent argument.

The cobounds are communicated to ASTI through the BUILDCO SYMs in W-Code++.

ASTI recognizes the BIFs and uses the cobounds information from the CoDescriptor in HIL to replace the BIFs with expressions that calculate the result.
