Feature Overview

This feature will implement the CRITICAL construct as specified in the Fortran 2008 standard.

CRITICAL was added to the Fortran 2008 standard, as part of the coarray feature. Its intent is to have native support for critical sections within the Fortran language.

Key Requirements

This feature is necessitated by the Fortran 2008 language standard and XL Fortran's continued conformance with this standard. It will define and implement a method by which a user can specify a critical section within the Fortran language. Such that the block within the section will always be executed by one image before or after another image.

Key Design Issues

  1. Communication of CRITICAL construct between the front end and later components.

Key Design Decisions

  1. Two directives will be generated delimiting a block that will only be allowed to be executed by a single image at a time. See Inter-component Communication for details.

Testing Strategy

Diagnostic testing should be performed that to ensure that none of the constraints below are broken. Functional testing should be written that ensure that only one image is executing a critical construct at a given time.

User Documentation

Overview

This feature will supply a new syntax CRITICAL / END CRITICAL to the user to allow them to define a critical section within Fortran source. To make use of the new syntax, the user must compile their source with the -qcaf command line option. Otherwise, CRITICAL / END CRITICAL will have no effect and will be ignored by the compiler.

User Benefits

This feature will allow the user to define a block of code that the compiler will guarantee that once one image has begun execution of that block, no other image will begin executing that block until the first image has completed executing the block.

If image T is the next to execute the construct after image M, the execution of the block on image M precedes the execution of the block on image T.

Public Interface

The CRITICAL construct can be used by the user using the following syntax:

  >---+----------------------------+--- CRITICAL ---><
      |                            |
      +---  construct_name  :   ---+
     
      ! ... Block
 
  >------- END CRITICAL ---+----------------------+---><
                           |                      |
                           +--- construct_name ---+


Constraints

  1. Transfer into or out of a CRITICAL construct is illegal. This includes the use of any of the following statements:
    1. Branches where the branch or branch target is inside the construct and the other is outside. (computed if, goto)
    2. EXIT
    3. CYCLE
    4. RETURN
  2. The CRITICAL and END CRITICAL statements both must have a construct name or both must not have a construct name. If construct names are present, they must match.
  3. A CRITICAL construct must not contain image control statements:
    1. SYNC ALL
    2. SYNC IMAGES
    3. SYNC MEMORY
    4. ALLOCATE or DEALLOCATE with a coarray actual argument
    5. CRITICAL / END CRITICAL
    6. LOCK / UNLOCK
    7. STOP
    8. END of main program
    9. any statement that completes execution of a block or procedure that results in implicit deallocation of a coarray
  4. To make use of CRITICAL / END CRITICAL the -qcaf option must be specified.

Rules

  1. Any procedure called from a CRITICAL construct must not contain an image control statement or cause the execution of the program to exit the CRITICAL construct.

Language Standard Considerations

Use of CRITICAL / END CRITICAL within a program compiled with a command line argument of -qlanglvl less than F2008 will result in a language level message being emitted.

Messages

Approved by the XLF Messages Committee.

Performance Considerations

The use of CRITICAL constructs will reduce the runtime performance of parallelized code caused by blocked images. Images will be blocked before a critical construct until it is an image's turn to execute that critical construct.

Samples

 
 function buyapples (count)
   integer :: buyapples
   integer :: count
   count = MAX(count,0)
   CRITICAL
     buyapples = MIN(remainingapples, count)
     remainingapples = remainingapples - buyapples
   END CRITICAL
 end function buyapples

High-Level Implementation Design

Implementation Design Overview

The FE will generate a symbol for each CRITICAL construct that will be used as the name of the PGAS RT lock for that particular CRITICAL construct. TPO will generate code to initialize these locks in the initialization module before the MAIN function is executed. The FE will also generate a pair of directives to indicate the beginning and end of the construct and TPO will generate calls to the PGAS RT to lock and unlock respectively.

Inter-Component Communication/Interfaces

The name of the lock variable should be __xlcaf_critical_N where:
  • N is a global counter that is reset in each compilation unit.
  • mt=static
  • owner=containing_procedure
  • dt=aggr
  • size=4 (for now. might change later)

Directives:

   DIR dir_no=caf_cs_start lock_var=<id_no of lock variable>
   DIR dir_no=caf_cs_end   lock_var=<id_no of lock variable>

Example:

   subroutine sub
   AAAA: CRITICAL
         ....
         END CRITICAL AAAA
   end subroutine
 
   SYM  id_no=3 len=4 owner=sub sym_flags=( ) flags=( )
        dt=aggr mt=static align=0 seg_no=0 nlen=18
        name=0x88{"__xlcaf_critical_1"};
      ...
   DIR  dir_no=caf_cs_start arg=3;
      ...
   DIR  dir_no=caf_cs_end arg=3;
