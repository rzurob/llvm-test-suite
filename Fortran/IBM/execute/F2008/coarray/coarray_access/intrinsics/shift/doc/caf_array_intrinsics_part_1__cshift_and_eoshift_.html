Overview

%Note{Simple description of the feature; In which product it is first implemented, is it common for other platforms, is it common for both C and C++ and so on.}%

This document describes testing of the two Fortran array intrinsic functions CSHIFT and EOSHIFT in a CAF (Coarray Fortran, which has now become part of the Fortran 2008 standard) environment. CSHIFT and EOSHIFT do a 'circular' and 'end of shift' of the elements of an array respectively. For detail description of these intrinsic functions refer to the Fortran 2008 standard [1]. Here is a very brief overview of CAF to understand the concept of image in CAF. For detail introduction on CAF read [2]:

A co-array in a Fortran program is replicated a fix number of times and all copies are executed asynchronously. Each copy has its own set of data and is called an image. The data in one image can be accessed in other image(s).

Test Strategy

%Note{Test strategy used in the test}%

This feature does not require an I0 document. The purpose of the testing in this feature is not to perform extensive testing of these two functions but to use these functions to test CAF. The testing will include using co-array with different types of data like integer, real and complex. Following is a list of some considerations when writing the new testcases (TCs):

  1. The test should be able to handle any number of images.
  2. Different ranks of array and co-array will be used in testing.
  3. One image preferably the first image to be used for initializing the data and then the initialized array to be communicated to other images.
  4. Data should be different on each image taken into consideration the image number. For example adding the image number to each element in the array will make sure that data on each image is different.
  5. We are not testing the functions itself so only one method of verification of data will be used. This is how the data will be verified:
    • For CSHIFT:
      • Store the original array.
      • Cshift the original array and then compare the results with the already known shifted array values.
      • Cshift the original array so that the new shifted array is equal to the original array and compare the shifted array with the original array bit by bit.
        • The data on images, other than the first image, can be compared with the data in the first image.
    • For EOSHIFT:
      • Eoshift the array so that all the elements in the shifted array are equal to the boundary value set while shifting.
      • Compare elements of the shifted array with the boundary value bit by bit.

Test Procedures/Scenarios

%Note{List and brief descriptions of the test cases created for the feature}%

The TCs are located at:

/tstdev/F2008/coarray/coarray_access/intrinsics/shift/


TCs Description Results
cshiftCoarray01.scenario Tests the CSHIFT intrinsic procedure as described above in the test strategy. It tests data of type integer, real and complex each of rank one and three. To avoid race condition risk, the origiinal stored value, which is not changed, is compared. The results are verified bit by bit by comparing it with the original stored array in image 1, and the execution is stopped on error using ERROR STOP
eoshiftCoarray01.scenario Tests the EOSHIFT intrinsic procedure as described above in the test strategy. It tests data of type integer, real and complex each of rank one and three.
 The results are verified bit by bit and the execution is stopped on error using ERROR STOP
cshiftCoarray02.scenario Tests the CSHIFT intrinsic procedure as described above in the test strategy. It tests data of type integer, real and complex each of rank one and two. Also use different co-array ranks. The results are verified bit by bit with the already known value shifted array and the execution is stopped on error using ERROR STOP
eoshiftCoarray02.scenario Tests the EOSHIFT intrinsic procedure as described above in the test strategy. It tests data of type integer, real and complex each of rank one and three. Also use different co-array ranks. The results are verified bit by bit and the execution is stopped on error using ERROR STOP
cshiftCoarray03.scenario Tests the CSHIFT intrinsic procedure as described above in the test strategy. It tests data of type integer, real and complex each of rank one and two. To avoid race condition risk, using SYNC IMAGES(*) so that image 1 can update the data before read by others and SYNC IMAGES(1) so that all other images wait for image 1 before reading the data.
 The results are verified bit by bit and the execution is stopped on error using ERROR STOP
eoshiftCoarray03.scenario Tests the EOSHIFT intrinsic procedure as described above in the test strategy. It tests data of type integer, real and complex each of rank one and three. Also use the image number to initialize the data. The results are verified bit by bit and the execution is stopped on error using ERROR STOP
cshiftCoarray04.scenario Tests the CSHIFT intrinsic procedure as described above in the test strategy. It tests data of type complex of rank one. First it initializes the data to 0 in all images. Then use image 1 to initialize the data. This data is then use by all other images to initialize there data. SYNC IMAGES(1) is used to make sure that the data is first initialized by image 1. After cshifting the data its compared with the knwon value stored in image 1.
 The results are verified bit by bit and the execution is stopped on error using ERROR STOP. Verification file is used to verify the initialization of data and results for image 1.
eoshiftCoarray04.scenario Tests the EOSHIFT intrinsic procedure as described above in the test strategy. It tests data of type complex of rank one. Also use the image number to initialize the data. First it initializes the data to 0 in all images. Then use image 1 to initializes the boundary data. This data is then use by all other images to shift. SYNC IMAGES(1) is used to make sure that the data is first initialized by image 1.
 The results are verified bit by bit and the execution is stopped on error using ERROR STOP. Verification file is used to verify the initialization of data and results for image 1.
