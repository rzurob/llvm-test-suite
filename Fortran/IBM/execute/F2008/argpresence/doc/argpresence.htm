Access Change History

 Access Change History

Feature Overview
This feature will implement the relaxed restraints on actual argument presence as described in the Fortran 2008 standard.

Key Requirements

In the Fortran 2008 standard, the following exception to argument presence:

A dummy argument or an entity that is host associated with a dummy argument is not present if the dummy argument does not have the ALLOCATABLE or POINTER attribute, and corresponds to an actual argument that has the ALLOCATABLE attribute and is not allocated, or has the POINTER attribute and is disassociated.
This feature must generate new W-Code to handle this exception.

Key Design Issues

1. This new functionality will introduce negative performance impacts due to the runtime check needed for whether the allocatable is allocated or the pointer is associated.
2. As the code generation is emitted on the caller side, the attributes, specifically optional, on the dummy argument may not be known.
3. As ASTI does not consume the SEL instruction, how can we communicate legal W-Code++ to ASTI that states it must choose to between two actual arguments

Key Design Decisions

1. A new option will be introduced to enable this behaviour. This option will default off if the compiler is invoked with a stanza xlf2003 or less. It will default to on when xlf2008 is invoked. The exact option will be reviewed by the XL Options Review Board.
2. The new code generation testing for the allocation and association status of the actual argument will only be performed when the procedure has an interface.
3. Until the SEL instruction is implemented, this will be implemented in the form of an ASTI BIF named __supplyoptarg_cond(descriptor, flag)

Testing Strategy

This feature will be tested via unit testing. The unit tests will cover a variety of differing program units, include module, intrinsic, external and internal procedures, where the actual and dummy arguments are derived types, parameterized derived types, polymorphic types and all possible intrinsic types.

User Documentation

Overview

The -qxlf2008=checkpresence option* will provide the user the opportunity to make use of the new actual argument presence detection.

* Under review by the ORB.

User Benefits

This feature has the potential to guard the user from incorrect results. Previously when an unallocated or disassociated object is passed to an optional dummy argument the PRESENT() intrinsic would return true, however the value passed was undefined. With this feature the PRESENT() intrinsic will return the correct result of false.

Public Interface

A new option will be introduced named -qxlf2008 with a suboption of checkpresence. The -qxlf2008=checkpresence option will enable this feature.

Syntax:

Option:

  
   >>----  -qxlf2008  --+----------------------------+------><
                        |                            |
                        --  =  --+-- checkpresence --+
                                 |                   | 
                                 -- nocheckpresence --    

 @PROCESS:


@PROCESS XLF2008 ( suboption, suboption, ... )

Defaults: The default suboption depends on the invocation command that you specify.


For the f2008, xlf2008, or xlf2008_r command, the defaults are:
checkpresence

For all other invocation commands, the defaults are:
nocheckpresence

Constraints

None.

Rules

The new behaviour takes effect when:

  1. The actual argument is an allocatable or pointer,
  2. The called procedure has an interface,
  3. The dummy argument is not an allocatable nor a pointer, and
  4. The dummy argument has the optional attribute.
  5. -xlf2008=checkpresence is specified

Language Standard Considerations

The -qxlf2008=checkpresence behaviour conforms to the Fortran 2008 standard. The -qxlf2008=nocheckpresence behaviour conforms to previous Fortran standards.If a language level of less than Fortran 2008 is used with the argument presence enhancement enabled, a language level message will be emitted stating that the actual argument may be removed from the actual argument list if not allocated or disassociated.

Messages

Language level message described above.

Performance Considerations

When -qxlf2008=checkpresence is in effect, the dummy argument presence check will negatively impact the performance of the user's program due to the increased amount of runtime code needed to determine if the allocatable or pointer is allocated or associated, respectively. A user should use -qxlf2008=nocheckpresence command line option or a xlf, f77, xlf90, xlf95, xlf2003 stanza, to ensure to avoid these performance impacts.

Multi-Platform/Portability Considerations

None.

Migration/Co-Existence/Binary-Compatibility Considerations

The result of the PRESENT() intrinsic may differ if the containing code is called from that compiled with the v14.1 compiler versus code compiled with v13.1 or before. This is expected to be rare since the value of the dummy argument was undefined in Fortran 2003.

Debug Considerations

None.

NLS (Unicode) Considerations

None

Samples

 integer, pointer :: p1(:), p2(:)
 integer, target :: t1(2)
 integer, allocatable :: a1(:), a2(:) 
 p2 => t1
 allocate (a2(1))
 
 call sub(p1,p2,a1,a2) 
 contains
 subroutine sub(w,x,y,z)
 integer, optional :: w(:), x(:), y(:), z(:)
 
 print *, PRESENT(w), PRESENT(x), PRESENT(y), PRESENT(z) 
! Output: with -qxlf2008=checkpresence:    F T F T
 !         with -qxlf2008=nocheckpresence:  T T T T
 
 end subroutine
 end

Testing Considerations

The functional unit tests should cover:

- Procedures:

  • Module procedures
  • Intrinsic procedures that have optional dummy arguments
  • External procedures (with and without interfaces)
  • Internal procedures
  • Elemental procedures
- Procedure calls within other procedures

- Actual and dummy arguments:
  • Derived types
  • Parameterized derived types
  • Polymorphic types
  • All possible intrinsic types.
  • Allocatable or pointer component within scalar and array derived types
  • Function result that is an allocatable or pointer
  • NULL() intrinsic
  • Specified via argument keyword in a differing order

The diagnostic unit tests should cover:
  • Interaction between -qxlf2008 and -qlanglvl


High-Level Implementation Design

Implementation Design Overview

When the -qxlf2008=checkpresence is enabled and the user passes an allocatable or pointer as an actual argument to an optional dummy argument that is not an allocatable or pointer, the compiler will perform the following:

  1. If a language level less than or equal to Fortran 2003 is enabled, emit a langlvl message stating that the allocatable or pointer might be removed from the actual argument list.
  2. Test if the actual argument is the NULL() intrinsic, if so generate the bif __SUPPLYOPTARG.
  3. Load the descriptor of the actual argument
  4. Load the DSCR_IS_ALLOCATED or DSCR_IS_ASSOCIATED flag, as appropriate.
  5. Generate a call to ASTI bif __SUPPLYOPTARG_COND, which takes two arguments.

Inter-Component Communication/Interfaces

A new ASTI BIF named __SUPPLYOPTARG_COND will be created. The BIF takes two arguments a descriptor and a flags field:

void __supplyoptarg_cond(d-arg, flags)

This BIF will test the flag argument against the flags within the descriptor. If the result is non-zero, it will load the descriptor and set the presence of the actual argument to true. Otherwise it will pass null and set the presence of the actual argument to false.
