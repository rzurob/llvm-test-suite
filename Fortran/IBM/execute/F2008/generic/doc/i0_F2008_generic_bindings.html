17 Feature Overview

18 This feature is a continuation of the generic binding work done last year under feature 219934. 19 Currently, the compiler supports generic bindings for dtio generic specifications only. 20 This feature will add support for generic bindings for defined operators, defined assignment, and generic names.

21 User Documentation
  • 22 Overview

    23 A generic binding specifies a type-bound generic interface for its specific bindings. 24 This allows the user to specify generic bindings for operators, assignment, derived-type IO, and generic names. 25 Generic bindings appear in the type-bound procedure block of a derived type definition.

  • 26 User Benefits

    27 This feature provides better encapsulation and object-orientation by effectively linking a generic interface to a type.

  • 28 Public Interface

    29 The syntax for generic bindings was already covered in the OO I0, and the 11.1 LR draft. Here it is for context purposes:
    30 

    31 
32 >>-- GENERIC -------------------- :: -- generic-spec -- => binding-name-list --><
33                |             |
34                -- , PRIVATE --
35                |             |
36                -- , PUBLIC ---
    where generic-spec is a specification of one of the following:
    • 37 generic name
    • 38 defined operator
    • 39 defined assignment
    • 40 derived-type input/output generic specification

  • 41 Constraints
    • 42 C459: Within the same derived type, each generic binding must have the same accessibility as every other generic binding with the same generic specification. 43 This also means that generic bindings in a subtype must have the same accessibility as inherited generic bindings with the same generic specification.
    • 44 C460: Each binding name appearing in a generic binding of a given derived type must be the name of a specific binding of that type
    • 45 C461: If the generic binding does not specify a generic name, each of its specific bindings must have the passed-object dummy argument.
    • 46 C539: An access specification can be specified for generic bindings only in the specification part of a module.
    • 47 Generic bindings follow the same rules for determining if dummy arguments are distinguishable as generic interfaces. 48 Another feature will add full support for TKR-compatability checking for type parameters.
    • 49 C462, C463: Generic bindings follow the same syntax rules for defined operators and defined assignment as generic interfaces.
  • 50 Rules

    • 51 Unless otherwise specified, generic bindings are resolved using the same rules as generic interfaces.
    • 52 If a generic binding specified in a type definition has the same generic specification as an inherited binding, it extends the generic interface.
    • 53 If a generic interface is accessible in the same scope as a derived type object containing an accessible generic binding with the same generic specification as the generic interface, the effect is as if the generic interface and the generic binding are merged.
    • 54 A specific binding name can appear multiple times in generic bindings with the same generic binding specification.
    • 55 The same procedure can be bound to the same generic identifier multiple times using generic bindings, and generic interfaces.

  • 56 Language Standard Considerations

    57 This is a Fortran 2003 feature.

  • 58 Messages

    59 Will be dealt with in the I1.

  • 60 Performance Considerations

    61 Run-time impact: Similar to how a generic interface resolves to a specific procedure at compile-time, a generic binding resolves to a specific binding at compile-time. 62 Unlike a generic interface, however, the specific binding can be polymorphic. 63 So there is a cost of a polymorphic call when the specific binding is polymorphic.

    64 Compile-time impact: Because we have to deal with per-type generic bindings, and with the interaction between generic bindings and generic interfaces, it is more time-consuming in the FE to determine which generic dr to use.

  • 65 Multi-Platform/Portability Considerations

    66 All platforms.

  • 67 Migration/Co-Existence/Binary-Compatibility Considerations

    • 68 Generic bindings for defined operators, defined assignment, and dtio can extend existing generic interfaces.
    • 69 The VAX-style component separator (.) 70 cannot be used if the scope has access to a derived type with a generic defined operator binding. 71 The standard (%) separator should be used instead. 72 (Note that there is an existing limitation (from xlf.v7r11) in which the VAX-style component separator cannot be used if a scope has access to a generic defined operator interface with the same name as the one appearing between the separators.)

  • 73 Debug Considerations

    74 Similar to generic interfaces, we resolve a generic binding to a specific binding at compile time. 75 Therefore, as long as the debug information for the specific binding is available, we should be able to step into a generic call in the debugger. 76 That said, it will not be possible to evaluate a generic binding in the debugger using the expression evaluator. 77 (Which is the same constraint as for generic interfaces.)

  • 78 NLS (Unicode) Considerations

    79 N/A.

  • 80 Samples

    • 81 The following program illustrates defines a defined assignment generic binding and a defined operator (+) generic binding. 82 In the example, the defined assignment binding, and all specific bindings are private to the module. 83 The defined operator binding is public. 
      84 
85 module m
86   type dt
87     integer i
88   contains
89     private
90     !
91     ! 92 Specific bindings
93     !
94     procedure :: add_int
95     procedure :: add_real
96     procedure :: assign_int
97     !
98     ! 99 Generic bindings
100     !
101     ! 102 Defined operator +
103     generic, public :: operator(+) => add_int, add_real
104     ! 105 Defined assignment. 106  Note that the same specific binding
107     ! 108 can appear multiple times.
109     generic :: assignment(=) => assign_int, assign_int
110   end type dt
111 
112   type(dt) dta
113 contains
114   integer function add_int(a, b)
115     class(dt), intent(in) :: a
116     integer, intent(in) :: b
117     print *, "add_int: a%i =", a%i, " b=", b
118     add_int = a%i + b + 1
119   end function
120 
121   type(dt) function add_real(a, b)
122     class(dt), intent(in) :: a
123     real, intent(in) :: b
124     print *, "add_real: a%i =", a%i, " b =", b
125     add_real = a%i + int(b) + 3
126   end function
127 
128   subroutine assign_int(a, b)
129     class(dt), intent(out) :: a
130     integer, intent(in) :: b
131     print *, "assign_int: b =", b
132     a%i = b
133   end subroutine
134 
135   subroutine init()
136     print *, "init"
137     dta = 4  ! 138 defined assignment
139   end subroutine
140 end module m
141 
142 use m
143 integer i
144 call init()
145 ! 146 The following statement uses the defined + operator twice.
147 ! 148 The first instance resolves to add_real, while the second
149 ! 150 resolves to add_int.
151 i = (dta + 3.0) + 3
152 print *, i  ! 153 14
154 end
155 
156 ! 157 Output:
158  init
159  assign_int: b = 4
160  add_real: a%i = 4  b = 3.000000000
161  assign_int: b = 10
162  add_int: a%i = 10  b= 3

177 High-Level Implementation Design
  • 178 Implementation Design Overview

    179 A generic interface dr will be created in the derived type scope of the type containing the generic binding. 180 If the specific binding referenced in the generic binding is polymorphic, the dr of the specific binding is saved in the dictionary auxiliary area. 181 Otherwise the dr of the procedure referenced by the specific binding is saved. 182 This is different from generic interfaces where the dr of the procedure is always saved in the dictionary auxiliary area.

    183 Since multiple drs can now hold the generic interface information, a new procedure GetGenericDr is created to returned the correct generic dr. 184 e.g. For a binary defined operator, the generic dr could be the one in the current scope (if a generic defined operator interface was specified), in the scope of the LHS, or in the scope of the RHS. Note that means that we have to attempt to resolve the generic reference to know which dr to use. (As a result, GenericReference must not issue error messages.)
