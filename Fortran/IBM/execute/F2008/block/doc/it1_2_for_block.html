Overview

%Note{Simple description of the feature; In which product it is first implemented, is it common for other platforms, is it common for both C and C++ and so on.}%

The BLOCK construct allows executable statements to be grouped in a new scope, and has the form

R807 block-construct is [ block-construct-name : ] BLOCK
           [ specification-part ]
           block
        END BLOCK [ block-construct-name ]

where the specification-part can contain zero or more USE statements or declarations (IMPORT and IMPLICIT statements are not allowed in this context, just as COMMON, EQUIVALENCE, INTENT, NAMELIST, OPTIONAL, statement function, and VALUE statements are not allowed).
and
where block is zero or more executable-constructs, format-stmts, or data-stmts (entry-stmts are not permitted in this context) - essentially, anything which can appear in a DO block can appear in a BLOCK.

Statements of the form
SAVE /blk/
are not permitted.

As is typical for block-like statements, BLOCK may be preceded by a name and a colon. If it is, the corresponding END BLOCK must also specify that name.

Block constructs can appear anywhere an executable construct can appear (i.e., as part of an execution-part or execution-part-construct). Examples of their use include:
b: block
  real :: localVar
  localVar = 1.2
end block b
 and
if (condition) then
  block
    integer, save :: ivar = 0
    print *, 'last =', ivar
    ivar = newVal
  end block

end if


Types can be defined in the block, and types defined in a containing scope can be redefined.

Variables introduced in the specification part of the block construct are local to that construct, except where the only declaration in the block of the variable is in an ASYNCHRONOUS or VOLATILE statement, in which case it marks a variable external to the block as ASYNCHRONOUS or VOLATILE for the scope of the block only. No IMPLICIT statement is allowed in the block, so any implicit typing is determined by IMPLICIT statements in the containing scope, including any references to derived types. Variables which are implicitly declared via a permitted attribute statement other than ASYNCHRONOUS or VOLATILE have the block as their scope. Variables introduced without any specification have the same scope as the IMPLICIT statement. For example:

type foo
  real :: comp = 1.2
end type foo
implicit type(foo) (a)
block
  type foo
    character :: comp = 'x'
  end type foo
  dimension :: aArr(1) ! aArr is known only within the block, has type foo from the outer scope
   print *, aVar        ! declared implicitly with default init, aVar is known throughout the program
  aVar%comp = 2.1
  aArr = aVar
  aVar = foo()         ! oddly enough, this is illegal (see below)
end block
! we can legally reference aVar here, but not aArr
end

This shows the declaration of a user-defined type and its use in an IMPLICIT statement, as well as the redeclaration of that type name within the block. Note that the type of the variable "aVar" and the array "aArr" is the outer "foo" type, so this block outputs "1.2", and not "x". Note also that the structrure constructor "foo" above actually refers to the inner type, making the assignment statement illegal due to type incompatibility.

More examples can be found in the F2008 language spec.



Test Strategy

%Note{Test strategy used in the test}%

Testing interactions with CAF features  (e.g., coarrays local to blocks) will be done in a different feature. Also, no testing will be done with OMP, since the OMP specification does not include Fortran 2003, let alone Fortran 2008.

Otherwise, several types of functional tests should be applied, touching on one or more of each of the following issues:

  • blocks in different contexts (main, nested, internal procedure, module procedure, external procedure, internal to external, ..., in elemental, in pure, in recursive)
    • >10 tests
  • (re)definition of variable name, e.g.,
    REAL R
    R=1.1
    BLOCK
      INTEGER R, S ! redefines R and S (implicit declaration first encountered after block)
      ...
    END BLOCK
    S = 2.1
  • check type, allowed operations, value, mixed-mode arithmetic, parameters, host association, use association
  • value from block scope does not overwrite outer value
  • block-to-block: separate entities if defined locally
    • >15 tests (1-2 per context)
  • (re)definition of types, without parameters
    • >15 tests (1-2 per context)
  • implicit statement not permitted
  • implicit declarations in containing scope may still have an impact (see example above in the overview)
    2-3 tests
  • "augmentations"
    • within a block, we can add the volatile or the asynchronous attribute to a non-coarray variable declared outside the block, so that the reference is to the same variable, and it functions as asynchronous or volatile only within the block. Note: a functional test of VOLATILE is too unpredictable to attempt, so it is tested elsewhere by examining the listing file for correct variable markings; this is not applicable here, since a variable declared outside the block but marked VOLATILE (or ASYNCHRONOUS) only within the block does not have that property always, and will therefore not be so marked in the listing file. Tests of VOLATILE, if they are performed, will have to be based on examining the WCODE: that they are treated as volatile within the block, but not outside it. Tests of ASYNCHRONOUS will be based on how they are treated by input/output statements (i.e., whether or not they are permitted in an asynchronous I/O statement).
    • all other attributes introduce an entirely new variable
    • thus:
      INTEGER ::  foo, bar
      BLOCK
        volatile :: foo, bar
        dimension :: bar(1)
        ! integer "foo" declared outside block is marked as volatile
        ! integer "bar" declared outside block is unavailable inside block
        ! the name "bar" refers instead to a volatile real array with bounds (1:1)
    • 5 tests
  • other permitted non-typed declarations (allocatable, bind, codimension, parameter, pointer, protected, save, target)
    • (5-10 tests)
  • suggestions appearing in the I0:
    - default initialization occurs as expected upon entering the construct
    - pointer association status is as expected upon exiting construct
    - finalization occurs upon exiting construct
    - deallocation of unsaved allocatable objects occurs upon exiting construct
    - SAVE statements outside the construct do not affect local objects
    - automatic objects have correct length, bounds, etc. upon entering the construct
    - local objects with SAVE attribute retain their values upon exiting construct
    - the debugging of variables declared in the BLOCK construct in totalview, dbx and gdb
  • COMMON, EQUIVALENCE, IMPLICIT, INTENT, NAMELIST, OPTIONAL, statement function and VALUE statements are not allowed in the specification part of a BLOCK construct (1 each)
  • A common block name is not allowed in the saved entity list of a SAVE statement in a BLOCK construct. (1)
  • If a block construct name is specified in a BLOCK statement, the corresponding END BLOCK statement must also specify the same name.  If no block construct name is specified in the BLOCK statement, the corresponding END BLOCK statement must also not specify a construct name. (2)
  • Transfer of control to the interior of the BLOCK construct from the outside is not allowed, except for the return from a procedure call inside the construct. (1)
  • The VOLATILE attribute cannot be specified inside a BLOCK construct for a coarray object that is not a construct entity of that construct. (1)
  • A local variable of a BLOCK construct within a pure subprogram cannot have the SAVE or STATIC attribute. (1)
