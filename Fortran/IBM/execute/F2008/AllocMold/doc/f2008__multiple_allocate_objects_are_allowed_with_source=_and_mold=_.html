Description:
Constraint 631 from Fortran 2003:

C631 (R623) If SOURCE= appears, type-spec shall not appear and allocation-list shall contain only one allocate-object, which shall be type compatible (5.1.1.2) with source-expr .

has been replaced by the following constraint in Fortran 2008:

C638 (R626) Each allocate-object shall be type compatible (4.3.1.3) with source-expr . If SOURCE= appears, source-expr shall be a scalar or have the same rank as each allocate-object.

Requirements:
Add support for multiple allocate-objects with SOURCE= and MOLD=, as described in Fortran 2008.
Emit a langlvl message with -qlanglvl=2003std/2003pure when more than one allocate-object is specified with SOURCE=.

Test Strategy

%Note{This is the most important section of the document.

Detail the test methodology and coverage for the feature to be tested.

Test strategy is not to identify all possible defects under all conditions with the feature. In most cases, this is unrealistic.
It is to identify and report defects with the feature under specific conditions likely to be used by the customer.
As a tester, you are the first customer of the feature. So you, the tester, need to understand how a certain feature would be used in the real world and come up with a Test strategy to validate those use cases.

Use cases to be validated should be documented here.

Provide examples if they are needed to explain the test method or coverage.

If applicable, organize the coverage into diagnostic and functional sections.

Diagnostic test cases verify error handling of the feature and the corresponding messages that are produced. This would also include verifying that valid syntax does not produce any unexpected messages.
This type of testing usually fall under the umbrella of unit tests. The developer of the feature is expected to write fully automated unit tests, but this can be negotiated if it is unfeasible for the developer to do so.

Test cases that verify values are referred to as functional test cases.

For large features, there may not be resource/time to completely test the feature. Certain parts of the feature will be more important than others.
Prioritize the test coverage and clearly document what will be covered in testing and what will be omitted. Risk assessment on the parts omitted should also be documented. This needs to be reviewed and agreed to by the requirements owner(s).

If there is already coverage in existing test suites, explicitly document it here.

Often, a feature by itself will work fine, but when combined with existing functionality or other new features, it may not work correctly. So ensure this type of testing is covered in the test coverage. This implies that other new features being developed for the release should also be understood.

Backward compatibility, portability, and any other migration Issues should be considered. One of the main complaints from customers is migrating from release to release (or from one platform to another).

Document any dependancies or open issues with the feature, so these are not over-looked.}%

Functional Test Cases:

 1. multAllocSourceMold01.scenario

Abstract: Multiple allocate of intrinsic types from source

For every intrinsic type this test case tries allocating multiple pointers to the same source.
  1. For numbers the value of the source varies from infinite negative to infinite positive where there is at least one test case for zero, negative or positive number.
  2. For characters the empty string, number string, character string and special character string
  3. For logical type only the possible TRUE and FALSE values.

Afterwards, checking the intrinsic type attributes of a costumed user defined data type.

The test case is also testing if mold is successful -- This part is limited due to defect 112628


2. multAllocSourceMold02.scenario
Abstract: Multiple allocate arrays of intrinsic types from source

This test case is similar to multAllocateWithSource01 except that it allocate an multiple arrays of every intrinsic type.

The test case is also testing if an array of a costumed user defined data type with intrinsic type attributes with 3 element where the first element is the default value, has the right values after multiple allocation

Checking allocation of sub-arrays using mold or source and checking the expected values after allocation.


3. multAllocSourceMold03.scenario
Abstract: Multiple allocate polymorphic types

This test case Multiple allocate polymorphic objects from a chain of inheritance where ZERO (the root super class) contains only an integer and the every descendant add another intrinsic type as attributes
For the object attributes of every intrinsic type the test case assigns the following:
  1. For numbers the value of the varies from negative infinity to positive infinity where there is at least one test case for zero, negative or positive number.
  2. For characters the empty string, number string, character string and special character string -- This part is not complete due to defect 112510.
  3. For logical type only the possible TRUE and FALSE values.

FIVE (the last user defined data type) has an attribute of FOUR (the last descendant in the inheritance chain).


4. allocateTests01.scenario
Abstract: Multiple allocating polymorphic objects

Trying to multiple allocate, allocatable target, class pointers and type pointers as well as allocatable arrays of 1d, 2d and 3d. Using source and mold.

The mold part might be wrong due to defect 112510. The verification file probably needs to be updated ones the defect is fixed.

All the AllocateWithSourceExpXX-08 test cases are a modified version of test cases from $(tsrcdir)/F2003/dtparam/allocate/SourceExp/AllocateWithSourceExpXX


5. AllocateWithSourceExp01-08.scenario
Test multiple allocation of arrays of the last polymorphic allocatable object of a chain of inheritance of user defined types containing arrays.

This is done directly as well as by calling a subroutine.

Also multiple allocate a pointer of the BASE class.


6. AllocateWithSourceExp02-08.scenario
Calling a subroutine to multiple allocate pointers of polymorphic type that has a pointer to type of its parent as an attribute.

** Where the source is one of the objects that are passed as arguments to the subroutine


7. AllocateWithSourceExp03-08.scenario
Multiple allocating a pointer of polymorphic type where the source is another pointer returned from a function


8. AllocateWithSourceExp04-08.scenario
Calling a subroutine to multiple allocate a polymorphic object that has a pointer to type of its parent as an attribute.

** The objects are local to the subroutine but the source is an allocated pointer passed as an argument


9. AllocateWithSourceExp05-08.scenario
  1. Calling a subroutine to multiple allocate pointers of the BASE type from a constructor.

  2. Calling a subroutine to multiple allocate the CHILD type using a constructed and a pointers as arguments to the constructor
 

10. AllocateWithSourceExp06-08.scenario
Multiple allocate pointers and objects of polymorphic types using a constructor then sending to a subroutine as argument that the subroutine uses as source to allocate again.

The CHILD type has 2 pointers of type BASE (its parent)

11. AllocateWithSourceExp06b-08.scenario
Similar to AllocateWithSourceExp06-08 but allocations in the subroutine are for multiple objects as well.

12. AllocateWithSourceExp07-08.scenario
Similar to but AllocateWithSourceExp06-08 this time the CHILD type has 2 allocatable objects of type BASE (its parent)


13. AllocateWithSourceExp08-08.scenario
  1. Multiple allocate pointers of CLASS(*) where the source is allocatable object BASE returned from a function.

  2. Multiple allocate pointers to array of CLASS(*) where the source is allocatable object BASE returned from a function.
 
 
14. AllocateWithSourceExp09-08.scenario
  1. Multiple allocating pointers of CLASS(*) with constructor of CHILD and pointer to parent as an attribute.
 
  2. Multiple allocating pointers of the CLASS(*) that are already allocated without deallocating where the source is another CLASS(*) pointer to CHILD object.


15. AllocateWithSourceExp10-08.scenario
  1. Multiple allocating pointers of defined BASE type from BASE constructor
 
  2. Taking one of the pointers (b1) and, without deallocating, assigning it the return of a function called SHUFFLE.
 
  SHUFFLE takes the other pointer (c1) as an argument and allocating it to the return.
  The function also take another argument of allocatable BASE object and multiple allocate it and a local allocatable BASE object using the BASE constructor.
 
  3. Taking the other pointer (c1) and associating with the return of a function.
  SHUFFLE takes the first pointer (b1) as an argument and allocating it to the return.
  The function also take another argument of allocatable BASE object and multiple allocate it and a local allocatable BASE object using the BASE constructor.
 
  4. Allocating allocating pointers of undefined BASE where the source is the function SHUFFLE.
 
  5. Allocating pointers of CLASS(*) where the source is the function SHUFFLE.


16. d339326.scenario
Calling a subroutine from a module with 3 arguments,
1 is N which size of the array and 2 are arrays of size N.
The subroutine multiple allocate the arrays using source=A(n)([(i, i=1, n)]))

17. AllocateWithMoldExp02-08.scenario
This AllocateWithSourceExp02-08.scenario converted to allocate with mold instead of source
Calling a function, sub2, check the type of the object and if it as expected it allocates the object with mold.
Checking the attributes of the allocated objects including the shape, size bounds of its arrays.
