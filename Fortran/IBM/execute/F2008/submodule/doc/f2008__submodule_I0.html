Access Change History

 Access Change History

Feature Overview

Submodule is introduced in Fortran 2008 standard. It provides additional structuring facilities for modules. A submodule is a program unit that extends a module or another submodule. A submodule may provide definitions for procedures whose interfaces are declared in an ancestor module or submodule. It may also contain declarations and definitions of other entities, which are accessible in its descendants. The declarations and the definitions of entities in a module or submodule are accessible by host association in its descendants.

Key Requirements


1. Implement the submodule feature so that it can provide definitions for procedures whose interface are declared in an ancestor module or submodule.
2. Implement the host association access of variables among module and its descendant submodules.

Key Design Issues


1. How to implement the host association access of a submodule and its ancestors as submodules and modules are all program units.
2. What file suffix should we use for the module symbol file created for a submodule.
3. How to handle the derived type promotion from submodule procedures scope into outer scope.

Key Design Decisions


1. The ancestor module and each of its descendant submodule will have its own scope. While the ancestor module possesses the outer most scope, a containing-contained scope relationship is established to achieve the host association.
2. Vendors don't need to ship the the module symbol files created for submodules as users only need to use a module. Therefore, we decided to use ".smod" file suffix for the module symbol files created for submodules to differentiate ".mod" for modules so that it is easy to filter them out.
3. We promote the use-associated derived types and all related internal symbols from the module procedure scope to its containing scope. Since the same module may be used by the ancestor module or submodules of a particular submodule, we need to search up to the ancestor module scope and create the symbol in the current submodule scope if it is not found.

Testing Strategy


There will be diagnostic and functional test cases covering the general use of submodule.

Positive diagnostic:
these will be scattered among functional test cases but involve:
- changing the lettering case of keywords such as 'subModuLe'
- changing the ordering of keywords such as 'module integer function f()' vs 'integer module function f()'

Negative diagnostic:
- the use of access modifiers for attributes in module functions
- function or subroutine defined in a submodule but not declared in any parent module
- function or subroutine defined in a submodule but declared in a parent module and a USE associated module
- other invalid uses of variables, procedures in scoping units

Functional:
- simple use declaring a submodule should pass
- checking variable accessibility in different scoping units

However, given that the code involving scoping units is higher risk, testing will be more heavily weighted towards scoping.  There are two ways scoping will be covered:

1.  Host association when extending a module with a submodule, which by default allows access to the host variables, type definitions, subroutine and function definitions

    The effect of access modifiers on variables, type definitions, functions and subroutines will be consired.  Test cases will consider the use of interfaces and also examine the name mangling of these functions and subroutines to verify the correct routine is called.

2.  Constructs that create a new scoping unit such as blocks, loops, USE association on a different module than the host, etc.

    These test cases consider the following:
    - scopes created by blocks, loops, etc are local to those blocks but can access host-associated items
    - with USE association on modules with conflicting attributes, which attribute is accessed and in which situation?
      - USE association can be set up within a local scoping unit or in the submodule

 

User Documentation

Overview

Submodule is introduced in Fortran 2008 standard. It provides additional structuring facilities for modules. A submodule is a program unit that extends a module or another submodule. A submodule may provide definitions for procedures whose interfaces are declared in an ancestor module or submodule. It may also contain declarations and definitions of other entities, which are accessible in its descendants. The declarations and the definitions of entities in a module or submodule are accessible by host association in its descendants.

User Benefits

The submodule feature provides an improvement in backward compatibility and usability. For example, when users use a module that contains the interfaces of module procedures, its descendant submodules may provide implementation of those module procedures. If vendors of those procedures change their implementation, as long as the interface remain the same, users are not required to recompile the files that contains the module.

Because submodules can not be accessed by use-association, submodules of two modules can use each other without causing the circular dependency.

Users can put entities in the intermediate submodule level that are shared by it descendant submodules. Changing these entities cannot change the interpretation of anything that is accessible from the module by use association. Users of modules that embody large complicated concepts can exploit this possibility to organize components of the concept into submodules, while preserving the privacy of entities that are shared by the submodules and that ought not to be exposed to users of the module. Putting these shared entities at an intermediate level also prevents cascades of reprocessing and testing if some of them are changed.

Public Interface


1. Syntax:
Submodule is:
submodule-stmt
[ specifiation-part ]
[ module-subprogram-part ]
end-submodule-stmt
submodule-stmt is:
SUBMODULE (parent-identifier) submodule-name
parent-identifier is:
ancestor-module-name [: parent-submodule-name]
end-submodule-stmt is:
END [ SUBMODULE [ submodule-name ] ]
2. Generated files:
ancestor-module-name_submodule-name.smod:
The submodule symbol file. It is used by its descendant submodules. The name of the file is the conjunction of the ancestor module name and the submodule name. The suffix of the file is ".smod" to allow users filtering them out easily. Unlike module symbol files (.mod files), the .smod files don't need to be shipped to the end users.
file_name.o:
The object file of a submodule. It contains the executable code of separate module procedures that are implemented in the submodule.

Constraints

The following constraints are straightly copied from the standard.

C1112: A submodule specification-part shall not contain a format-stmt , entry-stmt , or stmt-function-stmt. (same constraint as module)

C1113: The ancestor-module-name shall be the name of a nonintrinsic module; the parent-submodule-name shall be the name of a descendant of that module.

C1114: If a submodule-name appears in the end-submodule-stmt , it shall be identical to the one in the submodule-stmt.

C1506: A procedure defined in a submodule shall not have a binding label unless its interface is declared in the ancestor module.

C1247: MODULE shall appear only in the function-stmt or subroutine-stmt of a module subprogram or of a nonabstract interface body that is declared in the scoping unit of a module or submodule.

C517: An access-spec shall appear only in the specification-part of a module. (it is not valid in submodules).

Rules

1. A submodule is a program unit.

2. A submodule may contain module subprograms. It is the host scoping unit of the module subprograms.

3. A submodule may provide definitions for procedures whose interfaces are declared in an ancestor module or submodule. It may also contain declarations and definitions of other entities, which are accessible in its descendants. An entity declared in a submodule is not accessible by use association unless it is a module procedure whose interface is declared in the ancestor module.

4. A submodule has access to entities in its parent module of submodule by host association.

5. The end-submodule-stmt is nonexecutable.

6. A local variable of a submodule is accessible only in that submodule scoping unit and in any contained scoping units.

7. A variable, common block, or procedure pointer declared in the scoping unit of a submodule implicitly has the SAVE attribute, which may be confirmed by explicit specification.

8. If a specification expression in the specification-part of a submodule includes a reference to a generic entity, that generic entity shall have no specific procedures defined in the submodule subsequent to the specification expression.

9. If a constant expression in the specification-part of a submodule includes a reference to a generic entity, that generic entity shall have no specific procedures defined in the submodule subsequent to the constant expression.

10. A submodule has exactly one ancestor module and may optionally have one or more ancestor submodules.

11. The submodule identifier of a submodule is a global identifier.

12. Submodule identifiers are global identifiers, but because they consist of a module name and a descendant submodule name, the name of a submodule can be the same as the name of another submodule so long as they do not have the same ancestor module.

13. A separate module procedure shall not be defined more than once.

Language Standard Considerations

 SUBMODULE is a new feature in Fortran 2008 standard. A langlvl message is issued at the MODULE prefix keyword if the -qlanglvl option specifies older version of the standard.

Messages

Runtime and Compile Time Performance Considerations

Searching and reusing the symbols that have been use-associated in ancestor module or submodules will reduce the number of symbols. It will help the aliasing calculation so to improve the compile time performance.

Migration/Co-Existence/Binary-Compatibility Considerations

Debug Considerations

1. The name mangling scheme should be documented so that the debugger can demangle it.

2. Should the name of the submodule appear as "submodulename" or "(modulename)submodulename".

3. Submodule variables are invisible with Aix7 Dwarf.

Samples

 

Module m
  interface  
    module subroutine sub(arg) 
      integer arg
    end subroutine
  end interface
end module

submodule (m) n    
contains    
  module subroutine sub(arg)   
    integer arg  
    arg = 1
  end 
end submodule  
       
Program main     
  use m
  integer i      
  call sub(i)
end program

Testing Considerations


Since we use a containing scope chain to implement the host association among module and its submodules, the following test scenarios should be considered.

 1. Procedure pointer using procedures declared in different module/submodule scopes as its interface name or target.
2. Use statements in different module or submoduel procedures to test the derived type promotion.
3. Use modules in submodule scope to test the use-association vs the host association.
4. Test derived type I/O to make sure all the internal created I/O descriptors are in the correct scope.

High-Level Implementation Design

Implementation Design Overview


1. If a module procedure interface body is declared in a module, the mangled name of its corresponding separate module procedure and all the references to the separate module procedure is:
__NMOD_modulename_procedurename  // This is same as the existing name mangling schema for module procedures.
2. If a module procedure interface body is declared in a submodule, its corresponding separate module procedure, if there is one, and all the references to the separate module procedure must be in its descendant submodules. Therefore, the including the ancestor module name and the name of the submodule in which the module procedure interface body is declared makes a unique mangled name as:
__submodulename_NMOD_modulename_$procedurename
3. Same as a module, a submodule is also a program unit. We also create a hidden module procedure for debugging purpose for a submodule and mangle its name as:
__submodulename_NMOD_modulename_$_&&_submodulename
4. We also mangle the name of the container of submodule variables as:
&&N&submodulename_$_modulename      // for initialized submodule variables.
&&N&@submodulename_$_modulename  // for uninitialized submodule variables.
5. A containing scope chain is used to represent the host association. The outermost scope is the module scope. The current submodule scope is the innermost program unit scope.
