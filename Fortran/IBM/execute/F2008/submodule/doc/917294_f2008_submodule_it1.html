Overview

%Note{A brief description of the feature, its application(s) and limitation(s).

Feature syntax may also be mentioned here. Since it is in the I0 document, it is optional.}%

 

Submodule is introduced in Fortran 2008 standard. It provides additional structuring facilities for modules. A submodule is a program unit that extends a module or another submodule. A submodule may provide definitions for procedures whose interfaces are declared in an ancestor module or submodule. It may also contain declarations and definitions of other entities, which are accessible in its descendants.  The declarations and the definitions of entities in a module or submodule are accessible by host association in its descendants.  


Syntax:

Submodule is:
submodule-stmt
[ specifiation-part ]
[ module-subprogram-part ]
end-submodule-stmt
submodule-stmt is:
SUBMODULE (parent-identifier) submodule-name
parent-identifier is:
ancestor-module-name [: parent-submodule-name]
end-submodule-stmt is:
END [ SUBMODULE [ submodule-name ] ]

Constraints: 

The following constraints are straightly copied from the standard.

C1112: A submodule specification-part shall not contain a format-stmt , entry-stmt , or stmt-function-stmt. (same constraint as module)

C1113: The ancestor-module-name shall be the name of a nonintrinsic module; the parent-submodule-name shall be the name of a descendant of that module.

C1114: If a submodule-name appears in the end-submodule-stmt , it shall be identical to the one in the submodule-stmt.

C1506: A procedure defined in a submodule shall not have a binding label unless its interface is declared in the ancestor module.

C1247: MODULE shall appear only in the function-stmt or subroutine-stmt of a module subprogram or of a nonabstract interface body that is declared in the scoping unit of a module or submodule.

C517: An access-spec shall appear only in the specification-part of a module. (it is not valid in submodules).

Rules:

1. A submodule is a program unit.

2. A submodule may contain module subprograms. It is the host scoping unit of the module subprograms.

3. A submodule may provide definitions for procedures whose interfaces are declared in an ancestor module or submodule. It may also contain declarations and definitions of other entities, which are accessible in its descendants. An entity declared in a submodule is not accessible by use association unless it is a module procedure whose interface is declared in the ancestor module.

4. A submodule has access to entities in its parent module of submodule by host association.

5. The end-submodule-stmt is nonexecutable.

6. A local variable of a submodule is accessible only in that submodule scoping unit and in any contained scoping units.

7. A variable, common block, or procedure pointer declared in the scoping unit of a submodule implicitly has the SAVE attribute, which may be confirmed by explicit specification.

8. If a specification expression in the specification-part of a submodule includes a reference to a generic entity, that generic entity shall have no specific procedures defined in the submodule subsequent to the specification expression.

9. If a constant expression in the specification-part of a submodule includes a reference to a generic entity, that generic entity shall have no specific procedures defined in the submodule subsequent to the constant expression.

10. A submodule has exactly one ancestor module and may optionally have one or more ancestor submodules.

11. The submodule identifier of a submodule is a global identifier.

12. Submodule identifiers are global identifiers, but because they consist of a module name and a descendant submodule name, the name of a submodule can be the same as the name of another submodule so long as they do not have the same ancestor module.

13. A separate module procedure shall not be defined more than once.

Other notes:

In addition to the usual mod file generated at compile time, there will be smod files named ancestor-module-name_submodule-name.smod for each submodule.  This is the submodule symbol file. It is used by its descendant submodules. The name of the file is the conjunction of the ancestor module name and the submodule name. The suffix of the file is ".smod" to allow users filtering them out easily. Unlike module symbol files (.mod files), the .smod files don't need to be shipped to the end users.

 

Test Strategy

%Note{This is the most important section of the document.

Detail the test methodology and coverage for the feature to be tested.

Test strategy is not to identify all possible defects under all conditions with the feature. In most cases, this is unrealistic.
It is to identify and report defects with the feature under specific conditions likely to be used by the customer.
As a tester, you are the first customer of the feature. So you, the tester, need to understand how a certain feature would be used in the real world and come up with a Test strategy to validate those use cases.

Use cases to be validated should be documented here.

Provide examples if they are needed to explain the test method or coverage.

If applicable, organize the coverage into diagnostic and functional sections.

Diagnostic test cases verify error handling of the feature and the corresponding messages that are produced. This would also include verifying that valid syntax does not produce any unexpected messages.
This type of testing usually fall under the umbrella of unit tests. The developer of the feature is expected to write fully automated unit tests, but this can be negotiated if it is unfeasible for the developer to do so.

Test cases that verify values are referred to as functional test cases.

For large features, there may not be resource/time to completely test the feature. Certain parts of the feature will be more important than others.
Prioritize the test coverage and clearly document what will be covered in testing and what will be omitted. Risk assessment on the parts omitted should also be documented. This needs to be reviewed and agreed to by the requirements owner(s).

If there is already coverage in existing test suites, explicitly document it here.

Often, a feature by itself will work fine, but when combined with existing functionality or other new features, it may not work correctly. So ensure this type of testing is covered in the test coverage. This implies that other new features being developed for the release should also be understood.

Backward compatibility, portability, and any other migration Issues should be considered. One of the main complaints from customers is migrating from release to release (or from one platform to another).

Document any dependancies or open issues with the feature, so these are not over-looked.}%




Most functional testing will verify that the behaviour of various constructs in a module is similar to the same constructs in a submodule.  Likewise for declarations and definitions, which may also have access or storage modifiers as they would in a module, such as private, protected, public, allocatable.  These are expected to behave similarly in a submodule scope, when declared in the host.

There will be diagnostic and functional test cases covering the general use of submodule.

Positive diagnostic:
these will be scattered among functional test cases but involve:
- changing the lettering case of keywords such as 'subModuLe'
- changing the ordering of keywords such as 'module integer function f()' vs 'integer module function f()'
- confirm that smod and mod files are generated with the expected naming conventions
- confirm that a chain of submodule definitions are valid (covers different levels of depth and breadth)
Negative diagnostic:
- the use of a submodule-name that appears in the end-submodule-stmt and is different than the one in the submodule-stmt
- the parent-submodule-name is not the name of the parent submodule
- the ancestor-module-name is an intrinsic module
- the specification-part contains a format-stmt, entry-stmt, or stmt-function-stmt
- MODULE appears in the function-stmt or subroutine-stmt of a module subprogram or a nonabstract interface body that is declared in the scoping unit of a module or submodule
- the presence of a naming conflict between the host module and a USE associated module
- use-association of a submodule
- reusing the name of the host module or a submodule in another descendant submodule
 - the use of access modifiers for attributes in module functions
- function or subroutine defined in a submodule but not declared in any parent module
- function or subroutine defined in different way in a submodule and its parent module
- function or subroutine defined in a submodule but declared in a parent module and a USE associated module
- other invalid uses of variables, procedures in scoping units

Functional:
 - simple use declaring a submodule should pass
- use association of a module and its descendant submodules in a module with descendant submodules, which in turn may be use associated in another construct
- declaring submodules in different files
- define a module m, which uses module n, and a submodule nsub which extends n and uses m.  This should not cause a circular reference.
- submodule implementation decided on link time - i.e. only recompile a submodule without recompiling the rest of the source code.  This mimics the abstraction of library implementations, the main use of submodules.
- use of DTIO and procedure pointer. i.e. declaring in module and access it in different level of descendant submodules.
- checking public, protected, private variable accessibility in different scoping units (as well as parameter, allocatable, target, pointer, etc)

However, given that the code involving scoping units is higher risk, testing will be more heavily weighted towards scoping.  Unit testing will cover the private and protected variable scoping.

There are two ways scoping will be covered:

1.  Host association when extending a module with a submodule, which by default allows access to the host variables, type definitions, subroutine and function definitions (including those declared private, protected)

    The effect of access modifiers on variables, type definitions, functions and subroutines will be considered.  Test cases will consider the use of interfaces and also examine the name mangling of these functions and subroutines to verify the correct routine is called.

2.  Constructs that create a new scoping unit such as blocks, loops, USE association on a different module than the host, etc.

    These test cases consider the following:
    - scopes created by blocks, loops, etc are local to those blocks but can access host-associated items
    - with USE association on modules with conflicting attributes, which attribute is accessed and in which situation?
      - USE association can be set up within a local scoping unit or in the submodule


Examples of constructs considered:

- functions and subroutines
   - as a local scoping unit in a submodule, if not declared in the host interface interface
  - in an interface
      - use associate an external module in the interface in the host module vs the module subroutine
      - interface specified in a descendant submodule as a local scoping (unit only accessible and defined in its descendant submodules)

- derived type
   - as private type (should be accessible in descendant submodule, covered in UT)
   - as public type, but with private variables (same)
   - as private type with private variables (same)
  - pointer
      - procedure pointer in a derived type to a procedure in a host module or descendant submodule
      - to target types defined in or allocated in different scopes
    - generic pointer to a module procedure (candidate procedure may be defined in different submodules)

- do loop, OMP parallel
  - effects of shared, private variables
  - in local procedures, module procedures

- enumerator defined in the host module, accessible in descendant submodules
- block as a local scoping unit in a submodule, module procedure, nested procedure, etc.

The use of access modifiers will be covered primarily by unit testing.
