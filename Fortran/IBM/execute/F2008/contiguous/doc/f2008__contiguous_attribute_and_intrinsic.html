Feature Overview

This feature will implement the CONTIGUOUS attribute, CONTIGUOUS attribute specification statement and IS_CONTIGUOUS intrinsic function as specified in the Fortran 2008 language standard.

The Fortran 2008 language standard defines contiguous as:
- An array having array elements in order that are not separated by other data objects;

The Fortran 2008 language standard defines simply contiguous as the following:

6.5.4 Simply contiguous array designators
  1. A section-subscript-list specifies a simply contiguous section if and only if it does not have a vector subscript and
    1. all but the last subscript-triplet is a colon,
    2. the last subscript-triplet does not have a stride, and
    3. no subscript-triplet is preceded by a section-subscript that is a subscript.
  2. An array designator is simply contiguous if and only if it is
    1. an object-name that has the CONTIGUOUS attribute,
    2. an object-name that is not a pointer or assumed-shape,
    3. a structure-component whose final part-name is an array and that either has the CONTIGUOUS attribute or is not a pointer, or
    4. an array section
      1. that is not a complex-part-designator,
      2. that does not have a substring-range,
      3. whose final part-ref has nonzero rank,
      4. whose rightmost part-name has the CONTIGUOUS attribute or is neither assumed-shape nor a pointer, and
      5. which either does not have a section-subscript-list, or has a section-subscript-list which specifies a simply contiguous section.
  3. An array variable is simply contiguous if and only if it is a simply contiguous array designator or a reference to a function that returns a pointer with the CONTIGUOUS attribute.
      1. NOTE 6.16: Array sections that are simply contiguous include column, plane, cube, and hypercube subobjects of a simply contiguous base object, for example:
        ARRAY1 (10:20, 3)   ! passes part of the third column of ARRAY1.
        X3D (:, i:j, 2)     ! passes part of the second plane of X3D (or the whole
                            ! plane if i==LBOUND(X3D,2) and j==UBOUND(X3D,2).
        Y5D (:, :, :, :, 7) ! passes the seventh hypercube of Y5D.

Within this document the term "simply contiguous" is referring to an array that is known to be contiguous at compile time.
Within this document the term "simply non-contiguous" is referring to an array that is known to be not contiguous at compile time.

Key Requirements

This feature is necessitated by the Fortran 2008 language standard and XL Fortran's continued conformance with this standard.

Key Design Issues

  1. The IS_CONTIGUOUS intrinsic function can be folded by the front end in some cases. For all other cases ASTI should generate code to determine contiguity at runtime. The extent of this must be determined.
  2. W-Code must be updated to communicate contiguous objects between components.
  3. Full diagnostic testing for this feature would be very extensive. It must be determined what the compiler should and should not diagnose.
  4. In section 5.3.7, the standard outlines six conditions that constitute an object guaranteed to be contiguous and four conditions that constitute an object guaranteed not to be contiguous. Anything that falls outside these conditions are compiler dependent. These compiler specific conditions for contiguity must be identified.
  5. When a non-contiguous array is passed to an assumed-shape array dummy argument with the CONTIGUOUS attribute, the compiler must ensure that the argument associated object is contiguous.

Key Design Decisions

  1. The Fortran front end will fold the IS_CONTIGUOUS intrinsic function that has an array designator argument that can be defined as simply contiguous or simply non-contiguous, except for array sections. In cases where it cannot be determined at compile-time that an array is or is not contiguous, FE will pass a BIF to ASTI which will get inlined accordingly. This includes coindexed and coarray objects.
  2. A flag will be added to the w-code SYM to communicate between each component that the array has the contiguous attribute or is simply contiguous.
  3. The XL Fortran compiler will only diagnose errors where constraints are known to be broken at compile time. For example, in a pointer assignment statement where the pointer has the contiguous attribute, the front end will only emit an error if the target is simply non-contiguous.
  4. Compiler dependent contiguity:
    1. Contiguous
      1. Objects containing a single component with compile-time padding or rta. The FE will attempt to determine if the object is contiguous by looking at the compile-time padding. If the padding is equal to 0, the object is contiguous. If there is runtime alignment it cannot be determined at compile time. Instead the FE  will generate the ASTI __is_contiguous BIF. ASTI will generate code to determine if the rta is equal to the size of the component. If so, it is contiguous.
        In the following example        , x%i has compile-time padding when the testcase is compiled with -qautodbl=dblpad, which means it is not contiguous, however without this option it is contiguous.
                 program p1
                 type dt
                   integer i
                 end type
                         type(dt) x(10)
                 print *,IS_CONTIGUOUS(x)
                 print *,IS_CONTIGUOUS(x%i)
                 end
                 $ xlf2003 t.f 2>/dev/null  && ./a.out
                          T
                          T
                         $ xlf2003 t.f -qautodbl=dblpad 2>/dev/null  && ./a.out
                  T
                  F

    2. Non-contiguous
      1. Arrays with vector subscripts
      2. Objects containing components with run-time length and compile-time padding
                 program p2
                             type t (k,j,l)
                   integer, kind :: k, j
                   integer, len :: l
                   integer(k)::x
                   integer(j)::y(l)
                 end type
                 type(t(4,8,0)), allocatable:: t1(:)
                 allocate(t1(2))   ! t1%x is non-contiguous
                 print *,IS_CONTIGUOUS(t1)   ! T
                 print *,IS_CONTIGUOUS(t1%x) ! F
                 end
  5. If the actual argument to an assumed-shape dummy argument with the contiguous attribute is not contiguous, the compiler is expected, by the Fortran 2008 standard, to create a contiguous temporary actual argument to be argument associated. Furthermore, the compiler must perform copy-in/copy-out. This is a deviation from the original implementation of this feature, as this subtly was missed during design. The subtly was in the term "effective argument" which means the entity which is argument associated and not the user-specified actual argument. The appropriate lines in the standard are:
     12.5.2.3.6: The entity that is argument associated with a dummy argument is called its e ffective argument.
    1.3.63: effective argument entity that is argument-associated with a dummy argument
     5.3.7.1: The CONTIGUOUS attribute specifies that an assumed-shape array can only be argument associated with a contiguous effective argument
             with interp F08/0061, modified to: "The CONTIGUOUS attribute species that an assumed-shape array is contiguous"
    12.5.2.7.1: Constraint 1241: The actual argument corresponding to a dummy pointer with the CONTIGUOUS attribute shall be simply contiguous.

Testing Strategy


Unit tests will include functional test cases of the IS_CONTIGUOUS intrinsic function, both folded by the Fortran front end and inlined by ASTI. It will also include diagnostic testing to ensure the proper use of the CONTIGUOUS attribute. Diagnostic testing will be limited to simply contiguous objects.

User Documentation

Overview

This feature will provide the user with a new attribute CONTIGUOUS for type declaration statements, which will guarantee that the array object will be stored in contiguous memory. A new intrinsic function will be available to the user with the interface IS_CONTIGUOUS ( ARRAY ), where given an array will return a boolean value specifying whether ARRAY is stored in contiguous memory.

User Benefits

This feature will allow the compiler to further optimize runtime performance and storage.

Public Interface

Attribute:

CONTIGUOUS
Purpose:
The CONTIGUOUS attribute specifies that an array pointer or an assumed-shape array will have array elements in order that are not separated by other data objects.
Syntax:

>>--- CONTIGUOUS ---+------+--- contiguous_array_list ----
                    |      |
                    +- :: -+

Intrinsic Function:

IS_CONTIGUOUS (ARRAY)
Description. Test contiguity of an array
Class. Inquiry function.
Argument. ARRAY may be of any type. It must be an array. If it is a pointer it must be associated.
Result Characteristics. Default logical scalar.
Result Value. The result has the value true if ARRAY is contiguous, and false otherwise.

Example. After the pointer assignment AP => TARGET (1:10:2); IS CONTIGUOUS (AP) has the value false.

Constraints

A severe error will be emitted at compile-time if any of the following constraints are known to be broken:

  • An entity with the CONTIGUOUS attribute must be an array pointer or an assumed-shape array.
  • If the dummy argument is an array coarray that has the CONTIGUOUS attribute or is not of assumed shape, the corresponding actual argument must be simply contiguous.
  • The actual argument of the IS_CONTIGUOUS intrinsic function must be an array.
  • If the actual argument of the C_LOC or LOC intrinsic functions is an array, it must be contiguous.
  • If an actual argument is a nonpointer array that has the ASYNCHRONOUS or VOLATILE
    attribute but is not simply contiguous, and the corresponding dummy argument has either the
    VOLATILE or ASYNCHRONOUS attribute, that dummy argument must be an assumed-shape array
    that does not have the CONTIGUOUS attribute.
  • If an actual argument is an array pointer that has the ASYNCHRONOUS or VOLATILE
    attribute but does not have the CONTIGUOUS attribute, and the corresponding dummy argument has
    either the VOLATILE or ASYNCHRONOUS attribute, that dummy argument must be an array pointer
    or an assumed-shape array that does not have the CONTIGUOUS attribute.
  • The actual argument corresponding to a dummy pointer with the CONTIGUOUS attribute must be simply contiguous.
  • If the pointer of a pointer assignment has CONTIGUOUS attribute, the target must be contiguous.
     

Rules

The constraints that cannot be tested at compile-time will not be diagnosed, however they must be followed in the runtime case.

Language Standard Considerations

The CONTIGUOUS attribute and IS_CONTIGUOUS intrinsic function are Fortran 2008 features and will be flagged by -qlanglvl=2003std and below.

Messages

Will be approved by the XLF Messages committee.

Performance Considerations

The CONTIGUOUS attribute will give all components the opportunity to optimize the generated code due to the guarantee that the object is found in contiguous memory. There is no known negative performance impact.

Multi-Platform/Portability Considerations

None. 

Migration/Co-Existence/Binary-Compatibility Considerations

Source must be compiled with a v14.1 compiler or greater to make use of the contiguous feature. Use-associated objects from modules compiled with v13.1 or earlier will not be marked as contiguous unless it can be determined to be simply contiguous.

Testing Considerations

Functional and diagnostic testing needs to be written to ensure that an object's contiguity corresponds to the standard, specifically the following sections: 5.3.7, 6.5.4, 7.2.2.3.7, 7.2.2.3.9, 12.5.2.7.1, 12.5.2.8.2.

Testing will include ensuring the proper behaviour of use-associated objects compiled in 13.1 modules with the IS_CONTIGUOUS intrinsic and constraints regarding simply contiguous objects. It will also ensure that IS_CONTIGUOUS is not allowed in constant expressions when the arguments are assumed-shape arrays or pointer arrays and that IS_CONTIGUOUS is allowed (and works) in specification expressions.


High-Level Implementation Design

Implementation Design Overview

The front end will parse the CONTIGUOUS attribute and set the contiguous flag fedic.sld.d_has_contig_attr within that objects dictionary entry and Ss_t.seman_st.attr.is_contiguous on the stack. Array expressions will have their contiguity evaluated as per the definition of simply contiguous and the contiguity flags Ss_t.seman_st.is_simply_contig, fedic.sld.d_is_simply_contig or fedic.sld.d_is_not_contig will be set appropriately. If a dictionary entry is created from an object on the stack, the contiguous flag will be propagated. Module read and write will be updated to communicate the CONTIGUOUS attribute, however simply contiguous will not be communicated in modules. The IS_CONTIGUOUS intrinsic function will be folded by the front end for all simply contiguous array objects. In all other cases, FE generates a call to the __is_contiguous BIF. ASTI will generate code to determine contiguity at runtime. In cases where the ARRAY argument of IS_CONTIGUOUS is an array section, FE should create a temporary pointer object, pointer assign it to the section and pass the temporary descriptor to the __is_contiguous BIF. ASTI will setup the RTDF_IS_CONTIGUOUS flag on the descriptor during the pointer assignment and then generates code in place of the __is_contiguous BIF to check the RTD flag. 

Inter-Component Communication/Interfaces

A flag called SYMA_FL_IS_CONTIGUOUS_SHADOW will be added to the Sym{A} code within W-Code++. This will communicate that the sym was declared with the contiguous attribute.
