Feature Overview

Key Requirements


Fortran 2008 standard allows optional type specifications for index variables in a FORALL statement/construct. The current feature work will add support for this extension to FORALL. The compiler will issue a language level message in cases when source program includes the extension but Fortran 2008 is not the language level in effect.

Key Design Issues

In the absence of any type specification in FORALL header, the compiler can declare the index variables in the innermost executable construct or scoping unit that encloses the FORALL statement/construct. However, when a type specification is included in the FORALL header, the corresponding index variables have to be declared within the scope of the FORALL statement/construct (i.e. not in an enclosing user-defined scope).

Key Design Decisions


When FORALL header includes a type specification, the frontend can declare the index variable(s) in one of the following ways.

  1. The frontend can create a new BLOCK enclosing the FORALL and declare the index variable(s) as local to the newly created block.
  2. The frontend can declare the index variable(s) in the procedure scope that contains the FORALL (similar to the case when Forall-header does not include a type-spec). The existing infrastructure will ensure that any FORALL index variable is visible only in the scope of the corresponding FORALL.

The feature team prefers solution 2 as it avoids generating an extra block per FORALL and it does not require major changes in the FORALL code-gen in the frontend.

Testing Strategy


The testing will be split into two groups, diagnostic and functional.

Diagnostic tests will verify that appropriate error messages are emitted when forall is given an invalid type specifier, such as a non-integer type.  Various other invalid uses are expected to generate suitable error messages.  A language level (L) message is also expected when a forall declaration is correctly used but with an invalid langlevel in effect.

Functional tests will include basic valid use cases as well as nesting of the forall construct.  It can be nested within itself, with do, omp, SE, TM, or in other constructs.  If general, where a forall loop was valid before the extension, it should be verified that the forall loop is still valid using the extension and an appropriate type specifier.

 
User Documentation

Overview

The users can include a type specification in the FORALL header for the corresponding index variables. The type specification must specify an integer type. The specification applies to all indices in the Forall-header.

User Benefits

The FORALL index variables require explicit declarations in the following cases.

  • The range of values for the index variable exceeds the range of default integer type.
  • IMPLICIT NONE is in effect.
The extension to the FORALL syntax allows users to declare the type of index variables within the scope of the FORALL statement/construct.

In case, no type specification is included in the FORALL header, the index name must not be the same as an accessible identifier unless the later is a common block name or a scalar variable name. A type specification allows users to reuse any visible name as an index variable.

Public Interface

    forall_header is ( [ type_spec :: ] forall_triplet_spec_list [, scalar_mask_expr ] )

Constraints

The type specification (if included) must specify an integer type.

Rules

None.

Language Standard Considerations

The feature is a Fortran 2008 specific extension to FORALL. The type-spec in the Forall-header will be flagged by a language level message in case language level F2003 or lower is in effect. 

Messages

The language level checking will require the following new message (approved by the XLF Message committee).

    The type specifier in the %1$s construct is not permitted by the %2$s standard.

Here, "FORALL" will be used as the substitution text for %1$s. The substitution text for "%2$s" will be determined by the language standard in effect.

No new message is needed for the FORALL statement case (will reuse an existing message).

Debug Considerations

For debug purposes, we consider FORALL as a single step. The index variable declared using a type-spec will not be visible in the debugger.

Samples

 




Example 1:

       implicit none 
       integer, parameter :: LIMIT = 60000
       integer(1) :: flag(LIMIT) 
       flag = 0 
       FORALL ( integer(4) ::  i = 2 : LIMIT : 2)
          flag(i) = 1
       END FORALL 
       print *, flag(LIMIT) 
       END

Example 2:

       implicit none
       integer, parameter :: LIMIT = 1000
       logical(1) :: A(LIMIT, LIMIT)
       integer(1) :: i

       A = 0

       ! In the following, both i and j are of type integer(4)
       FORALL ( integer(4) ::  i = 2 : LIMIT : 2, j = 2 : LIMIT : 2 )
         A(i, j) = 1
       END FORALL

       print *, A(LIMIT, LIMIT)

       END

Testing Considerations

Test cases that involve nesting with SE and TM will be specific to blue gene Q, if they are applicable.

Iterator variables are also expected to be local, so that a reused variable name will not cause the external variable to be modified.

Internal or Hidden Options

None.

High-Level Implementation Design

Implementation Design Overview

Frontend will update the lex processor and grammar rules to accept the extended FORALL syntax. The type specification in the FORALL header only affects the type of the index variables. There will be no major change in the Wcode++. No changes are expected for downstream components (i.e. ASTI, TPO, Tobey).
